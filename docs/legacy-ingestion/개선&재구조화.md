## 서술 ##
- 사용자가 직접 rules들을 제어를 하게 하는건 불필요하다. 사람에게 인지적 부하를 가하는건 금지 (해당 task에 승인할 건지에 대한건 물어볼 수 있음)
- 현재는 각 task와 관련된 rule들을 검색만 해서 나열 해주는 정도일 뿐, 의미있는 처리가 전혀 아니다.
- "입력값 검증" 이라는 task는 모든 업무 프로세스에 걸쳐서 나타날 수 있는 task이다. (계좌등록, 회원가입, 결제 승인 등등)
  "어떤 업무 프로세스"의 입력값 검증에서 활용이 될 Business Rule인지 전혀 알아볼 수가 없다.
- 벡터 임베딩 유사도 검색은 속도는 빠르지만, 정확도가 떨어짐
- 에이전틱 RAG를 섞어서 유사도 검색을 통해 도출된 결과를 갖고 노드들을 직접 탐색시켜 정확도를 높일 수 있음
- 즉, 포괄적 의미의 task 자체를 명확하게 명시시켜서 구체적으로 어떤 업무 처리에 쓰일 task 인지부터 분할이 되어야하나?
- BL의 부모노드까지 탐색했을 때, 그 정보들을 구할 수가 있나?


- 네비게이터에서는 BPM 프로세스 모델 목록 자체가 나와야함 (현재 PDF to BPM이 단일 프로세스만 그리나?)
- 각 목록은 트리구조로, 펼치면 task 목록 & Domain 용어 들이 나올 수는 있음.



## 피드백 & 질문 ##
- Vector 검색을 해서 가져오면 모든걸 다 가져올 것이다.
- 자동납부계좌/카드번호실시간인증처리공통모듈 이라는 문서를 갖고 만든 것인데, 프로세스 이름은 어디에 있지? (트리구조화)
- "자동이체 계좌 등록"에 해당 하는 "입력값 검증" task에 해당하는 rule 들만 나오는건가?
- 각 rules 들이 command, aggregate 어디에 붙을지에 대한 결정을 할 수가 없으며, BPMN 단계 자체에서 작은 레벨의  로직은 붙일 수가 없음.
- 가장 중요한건, "계좌 자동이제 등록"과 관련된 "입력값 검증에" 해당하는 rule들만 나와야한다.
- 현재 BL + Function + Module 까지 더해서 벡터 검색을 했는데도 위의 필요한 필터링이 안됀거 아닌가?
- 각 task 하나 클릭해서 패널을 열었을 때, 에이전트를 띄워서 "자동이체 계좌 등록"의 "입력값 검증"에 관련된 rule들 탐색을 시작합니다 라고 하면서 온톨로지 트리들을 찾아가는 과정을 띄우도록 해도됌.
- 속도보다 정확도가 훨씬 중요함
- 사람은 인지적 부하를 낮추기 위해서 개입을 최소화해야함.

- 기존의 온톨로지 구조는 갖추어져 있으니, Module별로 vector 검색을 하여 top k를 뽑아서 "자동이체 계좌"와 관련된 프로세스 모듈들이 뭐가 있냐 먼저 찾아봐라.
- 모듈이 나왔으면, 각 모듈 하위에 있는 것중에 "입력값 검증"에 관련된 BL들은 뭐가 있나 찾아봐라.
- 나온 BL들의 상위 노드들 까지 묶어서 어떤 function, module에 매달려있는지 쿼리하여 "자동이체 계좌 등록"의 "입력값 검증"에 해당하는 것은 뭐냐고 다시 LLM에게 물어보면 각 task 별로 1,2개의 매우 정확한 매핑 결과를 가져올 수 있을 것임.

## 진행해야 될 작업 분석 & 의도 (2026-04-21 정리)

> 회의 내용과 피드백을 근거로, **근본 방향 전환**이 필요한 작업들을 정리한다. "적은 비용으로의 개선"이 아니라 위 문서에서 지적한 방향으로 **구조 자체를 재편**하는 것이 목표.

### 0. 방향 전환 선언 (현재 vs 이후)

| 축 | 현재 파이프라인 | 이후 지향점 |
|---|---|---|
| 철학 | "빠르게 후보 많이 → score 로 거르기" | "천천히 도메인 따라 내려가며 소수만 남기기" |
| 매칭 | 벡터 유사도 1-pass + score threshold | 계층적 에이전틱 탐색 (module → BL → 부모 검증) |
| 우선순위 | 속도 | **정확도 >> 속도** (Task 당 5~10s 허용) |
| 사용자 | 수동 편집 / 매핑 조정 툴 다수 | **승인·거부 2개 버튼만**, 나머지 Debug |
| 컨텍스트 | Task 이름만으로 검색 | Process 도메인 맥락 필수 동행 |
| AI 노출 | 결과만 조용히 반환 | 에이전트 **사고 과정 스트리밍** |
| 그래프 | BPM 단일 프로세스, session meta 에만 정체성 | `BpmProcess` 노드로 명시 |

### 1. 근본 진단 — 왜 현재가 부정확한가

회의의 핵심 문장: *""입력값 검증"이라는 task는 모든 업무 프로세스(계좌등록, 회원가입, 결제승인)에 나타나는데 "어떤 업무 프로세스"의 입력값 검증인지 알 수 없다."*

원인을 2개로 쪼개면:

**원인 A — Process 정체성이 그래프에 없음**
- PDF 파일명 (e.g., "자동납부계좌/카드번호실시간인증처리공통모듈") 에 이미 프로세스 정체성이 박혀있음
- 이걸 `HybridSession` 메타로만 갖고 있고, 그래프 노드로 저장 안 함
- Task 이름 `입력값 검증` 이 "어느 프로세스의 입력값 검증" 인지를 disambiguate 할 축이 그래프에 존재하지 않음
- → **`BpmProcess` 노드 도입이 선결**

**원인 B — 매칭이 "Rule 전체 공간 단방향 유사도"**
- `BL + Function + Module` 을 embedding 에 섞어도 의미적 유사도의 천장이 낮음
- "계좌등록 입력값 검증" 과 "결제승인 입력값 검증" 은 코사인이 거의 같게 나옴 → 구분 불가
- embedding 은 *유사도*를 측정하지 *논리적 귀속* 을 측정하지 않음
- → **에이전트가 그래프를 실제 탐색하며 귀속을 판단** 해야 함

#### 2.B Phase 3 전면 재설계 — Hierarchical Agentic Retrieval

**의도**
- 회의 §제안(51-53): Module top-k → BL top-k → 부모노드 검증·LLM 재확인 의 3단계
- Candidate 공간을 단계적으로 축소해 noise 를 기하급수 감소
- Task 당 결과를 1-2 개로 수렴
- **(2026-04-21 추가)** Multi-process 세션에서 Step 3 만으로는 프로세스 간 오염 해소 불가 → **Step 4 Arbitrator 필요**

**설계 — 4 Step Pipeline**

```
[Input] BpmTask (속한 BpmProcess.domain_keywords + task.name + task.description)
         │
         ▼
┌────────────────────────────────────────────┐
│ Step 1: Module Retrieval      [per task]    │
│  · query = Process.name + Process.domain_  │
│            keywords + Task.name            │
│  · analyzer DB의 MODULE 노드                │
│    (name, summary) 에 대해 vector top-k    │
│  · 결과: 관련 모듈 ≤ 5 개                   │
└────────────────────────────────────────────┘
         │
         ▼
┌────────────────────────────────────────────┐
│ Step 2: BL Retrieval within Modules        │
│                                 [per task] │
│  · 이전 step 의 모듈들에 소속된 FUNCTION    │
│    → BL 만 대상                             │
│  · 기존 embedding_matcher 재활용하되        │
│    rule corpus 를 module-prefiltered 로     │
│  · 결과: Task 후보 BL ≤ 15 개 (bl_top_k=15) │
└────────────────────────────────────────────┘
         │
         ▼
┌────────────────────────────────────────────┐
│ Step 3: Per-process Agentic Validator       │
│                           [per task × LLM 1회] │
│  · 후보 BL 전체를 한 번에 묶어 판정         │
│    (with_structured_output)                 │
│  · Cypher 사전조회: 부모 노드 체인          │
│    (CALLS / HAS_FUNCTION) 를 프롬프트 주입 │
│  · 각 후보에 verdict={accept, reject} +     │
│    rationale + evidence_refs               │
│  · **accept 개수 제한 없음** — judgment    │
│    quality 로만 판정 (정당하면 모두 accept) │
│  · 안전장치 상한 per_task_cap=20           │
│  · 결과: 프로세스 독립적 per-task 후보 집합 │
└────────────────────────────────────────────┘
         │
         ▼ (세션 전체 accept 수집)
┌────────────────────────────────────────────┐
│ Step 4: Cross-Process Arbitrator            │
│                          [경쟁 rule 당 LLM 1회] │
│  · 같은 rule_id 가 N ≥ 2 process 에서       │
│    accept 된 경우에만 호출                  │
│  · 입력: rule (GWT + context) +             │
│    모든 경쟁 주장 (process × task ×        │
│            per-process rationale)           │
│  · 출력: {reject=true}  (cross-cutting     │
│          utility → 매핑 안 함)             │
│        OR {home_process_id, home_task_id,  │
│            rationale}                      │
│  · 1 rule → 0 또는 1 (process, task)       │
└────────────────────────────────────────────┘
         │
         ▼
  (BpmTask)-[:REALIZED_BY {
    rationale: "b000_main_proc 에서 a000 호출,
                a000 은 공통 입력 검증으로
                자동이체 계좌 등록의 입력값 검증에 해당",
    evidence_refs: [...],
    evidence_path: [module_fqn, ...],
    agent_verdict: "accept",
    method: "agentic"
  }]->(Rule)
```

> **Step 4 가 왜 필요한가 (현장 발견)**: Step 3 는 per-process 독립 실행이라 "같은 rule 이 자동납부 신청 / 해지 / 조회 모두의 입력값 검증에 맞다" 같은 contamination 을 탐지 못 함. 실측으로 14 rule × 2~5 process 오염 확인. Step 4 는 경쟁 주장 전체를 LLM 에게 보여주고 **"진짜 home" 한 곳만** 선택하거나 cross-cutting utility 로 reject.

#### 2.C Agent 탐색 과정 UX 스트리밍

**의도**
- 회의 §58: "AI 로 사람이 하는 것 같으면 안 씀". AI 가 일하고 있음을 visible 하게
- Task 패널 열 때 자동 실행, 사고 과정 표시 → 인지 부하 ↓, 신뢰 ↑

**설계**
- 신규 SSE 엔드포인트: `GET /api/ingest/hybrid/task/{task_id}/retrieve`
- Task Inspector 오픈 시 자동 구독 (캐시 있으면 캐시, 없으면 실행)
- 이벤트 시퀀스:
  ```
  AgentStart                { task_id, process_name }
  AgentStepModuleSearch     { query, candidates: [{name, score, summary}] }
  AgentStepBlSearch         { module, candidates: [{rule_id, title, score}] }
  AgentStepParentLookup     { rule_id, parent_chain: [fn, module, callers] }
  AgentStepDecision         { rule_id, verdict, rationale }
  AgentFinalMatches         { rules: [...], total_ms }
  ```
- Frontend Inspector 내 "**Agent Reasoning**" 섹션 — 각 step 을 Claude 채팅 스타일로 순차 스트리밍
- 완료 시 "**이 Task 에 위 N 개 rule 이 맞나요? [승인] [거부]**" 버튼 표시

**완료 기준**
- 사용자가 Task 하나 열면 10 초 내 근거 + 승인 버튼까지 도달

### A. `BpmProcess` 노드 스키마 확정 (§2.A 구현용)

#### A.0 현 파이프라인 진단 — "새로 만드는 게 아니라 잃어버린 정보를 복원하는 일"

> **중요**: 외부 extractor 는 이미 multi-process 분할을 지원한다. 우리 어댑터가 평탄화해서 버리고 있을 뿐이다. Phase 1 재설계의 본질은 *새 능력 추가* 가 아니라 *상실된 분할 정보 복원* 이다.

| 계층 | 현재 상태 | 조치 |
|---|---|---|
| 외부 A2A extractor (`pdf2bpmn`) | ✅ 하나의 PDF 에서 여러 `<bpmn:process>` 생성 가능 | 그대로 활용 — 외부 서비스는 건드리지 않음 |
| `a2a_adapter.py::adapt_a2a_result_to_skeleton` | ⚠️ 받은 multi-process 를 `_merge_bpmn_definitions` 로 **단일 BpmSkeleton 으로 병합** ("every task is rendered on the canvas" 주석) | **병합 제거** — 반환 타입을 `list[BpmSkeleton]` 또는 `ProcessBundle(processes: [...])` 로 변경. |
| `entity_extractor.py` (native fallback) | ❌ `_ExtractionResult = {actors, tasks}` 스키마, process 개념 자체 없음 | 스키마 확장: `processes: list[{name, domain_keywords, actors, tasks}]`. 단일 프로세스 케이스는 processes 길이 1 로 표현. |
| `save_bpm_skeleton` | ❌ Task/Actor/Sequence 만 저장, process 없음 | `save_bpm_processes(list[BpmProcess])` 로 교체. 각 Process 가 자기 actor/task/sequence 집합 소유. |
| Neo4j 라벨 | ❌ `BpmProcess` 노드 미존재 | 신규 라벨 + `(BpmProcess)-[:HAS_TASK]->(BpmTask)` |
| Navigator 트리 | ❌ 단일 Task 목록 | Process N개 root → 펼치면 Tasks + Domain Glossary |

**함의**:
- 단일 프로세스 문서를 처리하던 기존 동작은 "processes 길이 1" 인 경우로 자연스레 degrade → backward-compat 유지
- 외부 extractor 와 우리 native fallback 양쪽 모두 **같은 scheme (processes 리스트)** 로 출력하도록 통일되면, 이후 Phase 3 Agentic Retrieval 에서 Process-level 필터가 즉시 적용 가능
- 현재 테스트 PDF 는 단일 프로세스라 분할 효과는 실데이터 확인 후 검증 필요

#### A.1 스키마 정의


### B. analyzer MODULE 노드 스키마 (§2.B Step 1 구현용)

**현재 DB 실측 기준** (2026-04-21 확인):
```
MODULE properties:
  name              — string, e.g., "zapamcom10060"
  fqn               — string, e.g., "apascrt00010t02_20260406.zapamcom10060"
  summary           — string (한국어), ✅ 채워져 있음. Step 1 vector query 의 주 타겟
  package_name      — string, e.g., "apascrt00010t02_20260406"
  moduleStereotype  — string, e.g., "Service" / "Repository" / ... (DDD stereotype 힌트)
  type              — string, e.g., "FILE"
  created_at / updated_at — int
```

**주의**: 현재 테스트 데이터는 `MODULE` 1개 · `FILE` 1개 가 동일 내용 중복 저장. Step 1 구현 시:
- Primary: `(m:MODULE)` 대상으로 쿼리
- Fallback: `MODULE` 미존재 시 `(f:FILE)` 로 degrade — 두 라벨이 같은 속성 세트를 가짐

**Step 1 Cypher 템플릿**:
```cypher
MATCH (m:MODULE)
WHERE m.summary IS NOT NULL AND m.summary <> ''
RETURN m.name AS name,
       m.fqn AS fqn,
       m.summary AS summary,
       m.moduleStereotype AS stereotype
```
→ Python 쪽에서 vector 검색 (OpenAI `text-embedding-3-small`, Process.domain_keywords + Task.name 을 query 텍스트로) + top-k

**MODULE 임베딩 캐싱**:
- 소규모: 세션 in-memory `EmbeddingCache` (기존 사용 중)
- 중규모(>100 MODULE): `MODULE.embedding: list[float]` 프로퍼티로 pre-compute + Neo4j 저장
- 대규모: Neo4j 5.11+ `CREATE VECTOR INDEX` 로 인덱싱 (별도 인프라 불필요)

### C. Agent Validator — 프로젝트 LLM 패턴 맞추기 (§2.B Step 3 구현용)

**현 프로젝트 LLM 호출 스타일**: `with_structured_output(PydanticSchema)` — **tool-calling / ReAct / LangGraph 미사용**. 20+ 곳에서 동일 패턴으로 구현됨 (`workflow/phases/*.py`). 일관성을 위해 Step 3 도 이 패턴을 따른다.

**즉, "Agentic Validator" 는 자율 tool-calling agent 가 아니라 `(Cypher 사전조회) → (Structured LLM 판단)` 시퀀스**:

```python
# mapper/agent_validator.py (신규)
from pydantic import BaseModel, Field
from api.features.ingestion.ingestion_llm_runtime import get_llm

class _ValidationVerdict(BaseModel):
    rule_id: str
    verdict: str = Field(description="accept | reject")
    rationale: str = Field(description="1~2 문장 한국어, 근거 제시")
    evidence_refs: list[str] = Field(default_factory=list)

class _ValidationResult(BaseModel):
    verdicts: list[_ValidationVerdict]

async def validate_candidates(
    process_name: str,
    process_keywords: list[str],
    task_name: str,
    task_description: str,
    candidate_bls: list[CandidateBL],  # Step 2 출력
) -> list[_ValidationVerdict]:
    # Sub-step 1: 각 후보의 부모 노드 체인을 Cypher 로 미리 조회 (LLM 호출 아님)
    enriched = []
    for c in candidate_bls:
        chain = fetch_parent_chain_cypher(c.rule_id)  # callers / module / package
        enriched.append({**c.dict(), "parent_chain": chain})

    # Sub-step 2: 단일 LLM call 로 전체 후보 판정 (구조화 출력)
    prompt = build_validation_prompt(process_name, process_keywords,
                                      task_name, task_description, enriched)
    structured_llm = get_llm().with_structured_output(_ValidationResult)
    result = await structured_llm.ainvoke(prompt)
    return result.verdicts
```

**핵심 포인트**:
1. "Tool-calling" 을 시뮬레이션할 필요 X. 필요한 그래프 조회는 LLM 호출 *전에* Cypher 로 수행해서 텍스트로 promtp 에 주입.
2. 한 Task 당 LLM 호출은 **1회** 가 원칙 — 후보 BL 을 묶어 한 번에 판정. 비용·속도·일관성 모두 이득.
3. 결과 pydantic 스키마 는 기존 컨벤션 맞춰 `PascalCase` + `List` suffix (e.g., `_ValidationResult`).

**에이전트 "사고 과정" UX (§2.C)** 는 내부적으로 tool-call 로 진행되는 것처럼 보이면 됨 — 실제로는 Python 이 Cypher 수행 → SSE 로 진행 이벤트 발행 → LLM 최종 판정 → SSE 최종 이벤트. **LLM 이 직접 tool 을 부르지 않아도 UX 는 동일**.

### 8.6 구조적 교훈

1. **Task-level embedding 은 coarse-grained module summary 와 미스매치**
   → process-level max score 로만 gate 하고 개별 task 는 rank 로 정렬.
2. **Validator 는 per-call 결정이라 "다른 task 와 비교" 불가능**
   → sibling_tasks 컨텍스트 주입으로 비교 판단 가능하게.
3. **Arbitrator 는 module_confidence 수치에 취약**
   → stage-first 규칙 + 0.05 이하 수치 차이는 무시.
4. **rule.title 은 가장 강력한 business-intent 신호**
   → embedding blob 최상단에 배치, function_summary keyword bias 완화.
5. **SSE 재탐색은 batch 파이프라인과 파라미터가 달라야 안전**
   → gate 우회 + DELETE 순서 조정.
6. **프론트 auto-run 은 race condition 의 근원**
   → 명시적 사용자 액션(🔄)에만 매핑 변경 허용.
