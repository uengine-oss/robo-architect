## Q1. 컨테이너 모듈 게이트 — 제거 vs 약화

**실험**: 게이트(모듈 검색→프리필터) 없이, 각 task 질의 vs 전 rule(블롭 A) 코사인 → floor별 above-floor 후보 수.

| | floor 0.45 | floor 0.40 | floor 0.35 |
|---|---|---|---|
| 후보 수(task 평균) | **70** | 70.4 | 72.8 |
| **후보 0인 task** | **0 / 19** | 0/19 | 0/19 |

- 현재 코드(게이트 有)= 전 task **0건**(모듈 0개→프리필터 전탈락). 게이트 제거 시 **19/19 task 모두 후보 생성**(0 → 총 1330).
- **결정: 컨테이너 모듈 게이트를 하드 차단으로 쓰지 않는다(제거).** 모듈 요약 점수는 (필요 시) 로그/약신호로만. 약화(문턱만 낮춤)는 여전히 거친 모듈 요약에 의존 → 거대 단위에서 재발 위험. 제거가 단순하고 근본.
- **부수 고려**: 게이트가 하던 "이 프로세스에 대응 코드 없음 → 통째 스킵"은, 대응 코드 없는 프로세스에선 검증기가 후보를 전량 거부 → 자연히 0매핑(올바른 결과, 검증 LLM 비용만 약간↑). 교차오염은 기존 검증기/중재가 처리.

## Q2. Rule 임베딩 묶음(blob) 구성 — GWT만 vs ＋요약 vs ＋테이블

**실험**: 같은 task에 대해 blob 3변형으로 above-floor(0.45) 후보 수(평균).

| 변형 | 후보 수(평균) | 해석 |
|---|---|---|
| A: GWT(title+given/when/then)만 | **70 / 91** | 너무 넓음(거의 다 통과) |
| **B: A + 소속 루틴 summary** | **38 / 91** | **가장 선별적**(풀 절반) |
| C: A + 관련 테이블 | 69 / 91 | A와 무차이(테이블 44/91만 보유) |

- **결정: blob = GWT + 소속 루틴 summary (B).** 실제 배포 모델에서 풀이 가장 타이트(검증기 입력·비용↓). 테이블(C)은 커버리지 낮아 효과 미미 → 채택 안 함(선택 신호로 둘 수 있음).
- **★중요 정정 기록**: `agentic_retriever.py`의 lean-blob 주석("summary 포함 시 43개로 풀 과대 → GWT만 lean")은 **OpenAI text-embedding-3-small 기준**이었고, **현 배포 SGLang 모델에선 정반대**(summary 포함이 더 선별적). 즉 "summary를 빼라"는 근거는 모델 의존이며 본 환경엔 적용 안 됨. (헌법 III 단일진실: 모델 바뀌면 이 가정 재확인 필요 = 매직 가정 금지.)

## Q3. 규칙 폭증 시 비용 상한

**실험**: 91개 rule 블롭 임베딩 1회 소요.

- 단건 호출로 **3.4초 / 91개**(캐시·소배치면 더 빠름). 세션 규칙 수십~백 개 규모에서 **전수 임베딩 비용 무시 가능** → **하드 비용 상한 불필요.**
- **제약(실측)**: 임베딩 엔드포인트 **배치 상한 32** + 큰 배치 GPU OOM → 임베딩 호출은 **≤32 청크(권장 ≤30)** 로 분할해야 한다(현 `EmbeddingCache.embed_many`는 무제한 배치 → 청킹 보강 필요).
- **결정**: 비용 상한은 두지 않되, ① 임베딩을 ≤30 청크로 분할(견고성), ② 안전망으로만 세션 rule 수 soft-cap(예: 수천 초과 시 top-K 선별) 정도. 정밀도/recall 저하 없음.

## 추가 핵심 발견 — 임베딩 변별력 (설계에 영향)

- 이 배포 모델은 **동일 도메인 한국어 rule들을 0.79~0.82로 뭉쳐** 매김(샘플 top3 전부 0.79~0.82). 즉 **floor는 거의 필터링을 못 함**(70/91 통과), 같은 소수 rule이 여러 task의 top에 중복 등장 → **task 간 정밀 변별은 임베딩이 아니라 기존 LLM 검증기/중재가 담당**.
- 그럼에도 top 매칭은 **도메인상 타당**(예: "자동납부 신청서 접수" → top "기업은행 평생계좌 자동납부 / 간편결제 자동납부 신청 / 카드 자동납부 무인증"). 즉 **0건과는 천지차이**로, rule-grained가 "관련 풀"을 정확히 회복한다.
- **설계 함의**: 임베딩은 **거친 recall 프리필터 + top-K 상한**으로 쓰고, **정밀도는 기존 검증기/중재**에 맡긴다(현 파이프라인 설계와 합치). 본 피처는 "0 → 관련 풀 회복"이 목표이며 정밀도 장치는 그대로 재사용(spec FR-004).
