## Background & Root Cause *(왜 이렇게 하는가 — 라이브 검증 근거)*

> 이 절은 "왜 이 변경이 필요한가"를 사실 근거로 남긴다. 구현 방법(HOW)이 아니라 **현상·원인·왜 다른 단위여야 하는가**를 기록한다.

### 현상 (실측)
- 코드분석 모드(analyzer 그래프 + 업무 PDF → BPM → 전체탐색 → 이벤트스토밍)에서, 업무 단계(BpmTask)와 분석된 코드 규칙(Rule)을 잇는 매핑이 **0건**으로 나온다.
- 결과: 이벤트스토밍 그림은 그려지지만 **코드와 전혀 엮이지 않아** "코드분석" 모드의 핵심 가치(업무↔코드 추적성)가 사라진다.
- test 그래프(386노드, framework) 풀런 실측: Task↔Rule 매핑 0건, 추출된 Rule 91개 전부 고립(어떤 Task에도 안 붙음), BpmTask 19개 중 매핑 0개.

### 근본 원인 (코드·임베딩으로 추적)
- 매칭이 업무 Task를 **코드 "컨테이너(파일/모듈)"의 요약**과 비교한다. 컨테이너 하나가 여러 단위를 품으면 요약이 "이것저것 다 함" 식으로 두루뭉술해진다.
- 그 두루뭉술한 컨테이너 요약은, 구체적인 업무 Task("신청서 접수")와의 의미적 유사도가 통과 기준 미만으로 떨어진다 → 후보가 0이 되고, 그 컨테이너 안의 Rule 후보까지 전부 탈락한다 → 매핑 0.
- **실측 유사도**(이 C 그래프): 파일(컨테이너) 요약 vs Task ≈ 0.28~0.31(통과선 0.45 미달, 탈락). 개별 함수(루틴) 요약 vs Task ≈ 0.61(통과). 즉 컨테이너가 너무 거칠다.
- 이 코드는 framework 전략(규칙이 함수에 직접 부착, 구문트리·DML 노드 없음)이고, **파일 1개에 함수가 17~21개** 들어있어 컨테이너 요약이 뭉뚱그려진다.

### "루틴(함수/프로시저) 단위로 바꾸면 되는가" — 부분적
- C는 함수 단위로 바꾸면 통과(0.61). 그러나 **DBMS는 프로시저 하나가 구문 수십~수백 개를 품을 수 있어** 프로시저 요약도 파일처럼 거칠어질 수 있다 → 루틴 단위도 "거대 단위"에선 동일 문제가 재발할 수 있다.
- 즉 **컨테이너-단위도 루틴-단위도 "단위 크기"에 취약**하다. 크기와 무관하게 **항상 구체적인 유일한 비교 대상 = 개별 Rule**(코드분석이 이미 Given/When/Then으로 잘게 추출). Rule은 1000줄 프로시저 안에 있어도 그 자체로 구체적이다(예: "필수 필드 누락 시 거래 거부 코드 반환").

### 회귀가 아니라 원래 한계 (git 확인)
- 직전 작업(044)은 매칭 코드의 라벨·속성 **이름표만** 현행 analyzer 출력에 맞춰 바꿨다(컨테이너 라벨/식별자 갱신). **비교 "단위"(컨테이너/파일)는 최초 도입 때부터 동일**했다.
- 따라서 이 0건 문제는 044가 새로 만든 회귀가 아니라, **컨테이너-단위 매칭의 원래 한계**가 "파일당 다단위" 코드에서 드러난 것이다. (파일=단위1개인 코드에서는 우연히 잘 동작했다.)
- 044 권위 계약(C4)은 이미 "오퍼레이션/매칭 단위 = 루틴, 모듈(파일/클래스)은 단위 아님(보조 메타)"이라고 명문화했으나, 매칭 구현은 컨테이너로 게이팅해 이 계약을 따르지 않고 있었다. 본 피처는 이를 한 걸음 더 밀어 **Rule 단위**로 정합한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 코드분석 결과가 설계에 실제로 반영된다 (Priority: P1)

아키텍트가 업무 문서와 (이미 분석된) 소스코드를 코드분석 모드로 올리고 전체탐색을 돌리면, **각 업무 단계에 그 단계를 구현하는 코드 규칙이 매핑**되어, 이벤트스토밍·설계가 실제 코드에 근거하게 된다. (현재는 매핑이 0건이라 그림이 코드와 무관하다.)

**Why this priority**: 이 매핑이 "코드분석 모드"의 존재 이유다. 0건이면 기능 전체가 무의미하다. 따라서 최우선.

**Independent Test**: 코드분석 풀런 후, 코드가 구현하는 업무 단계들에 코드 규칙이 1건 이상 붙는지 그래프/화면으로 확인. 매핑 수가 0 → 양수로 회복하면 가치 입증.

**Acceptance Scenarios**:

1. **Given** 분석된 코드 그래프와 업무 문서가 인제스천되어 BPM·Rule이 만들어진 상태, **When** 전체탐색을 실행하면, **Then** 코드와 관련된 업무 단계들에 코드 규칙 매핑이 생성된다(매핑 총량 > 0).
2. **Given** 매핑이 생성된 상태, **When** 이벤트스토밍을 생성하면, **Then** 설계 산출물이 코드 규칙에 근거해 만들어진다(코드와 무관한 빈 껍데기가 아니다).

---

### User Story 2 - 코드 구조 크기와 무관하게 동작한다 (Priority: P2)

한 파일에 함수가 많은 코드(C 등)나, 한 프로시저가 거대한 코드(DBMS 등) 모두에서, **단위 크기에 상관없이** 업무 단계에 규칙이 매핑된다.

**Why this priority**: 실제 레거시(특히 DBMS)는 단위가 거대한 경우가 흔하다. 특정 코드 모양에서만 되는 매핑은 신뢰할 수 없다. P1 가치의 일반화이므로 P2.

**Independent Test**: 파일당 단위가 많은 그래프에서 매핑이 생성됨을 확인(현 framework 그래프). 거대 단위(DBMS 프로시저) 그래프 확보 시 동일 확인.

**Acceptance Scenarios**:

1. **Given** 파일 1개가 17~21개 함수를 품은 코드, **When** 전체탐색을 실행하면, **Then** 컨테이너 요약이 거칠더라도 규칙 매핑이 생성된다.
2. **Given** 한 프로시저가 다수의 구문을 품은 DBMS 코드, **When** 전체탐색을 실행하면, **Then** 프로시저가 거대해도 규칙 단위 비교로 매핑이 생성된다.

---

### User Story 3 - 부정확한 매핑은 걸러진다 (Priority: P3)

규칙을 더 많이/직접 비교하면서도, **명백히 관련 없는 규칙은 기존 검증·중재 단계가 걸러내** 매핑 품질을 유지한다.

**Why this priority**: 매핑을 늘리는 것이 오매핑을 늘리는 것이어선 안 된다. 정밀도 보호. 기존 안전장치 재사용이라 P3.

**Independent Test**: 매핑 결과에 명백한 오매핑이 도배되지 않는지(검증 거부/중재 동작 유지) 확인.

**Acceptance Scenarios**:

1. **Given** 한 업무 단계에 후보 규칙이 여럿 떠오른 상태, **When** 검증·중재를 거치면, **Then** 관련 없는 후보는 거부되고 합당한 매핑만 남는다.

---

### Edge Cases

- 코드와 **무관한 업무 단계**(대응 코드 없음): 매핑 0이 정상 — 억지 매핑을 만들지 않는다.
- **거대 단위**(파일당 함수 多 / 거대 프로시저): 단위 요약이 거칠어도 규칙 단위 비교로 후보가 살아남아야 한다.
- **규칙이 아예 없는 코드**: 후보 0 → 매핑 0(에러 아님).
- **여러 단계가 같은 규칙을 주장**(경합): 중재가 한 곳으로 정리한다(기존 동작 유지).
- **규칙 수가 비정상적으로 많은 세션**: 전수 비교 비용이 급증하지 않도록 합리적 상한을 유지(품질 저하 없이).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 업무 Task와 코드 Rule을 매핑할 때, **개별 Rule과 그 Rule의 컨텍스트**(예시 Given/When/Then, 소속 루틴 요약, 관련 테이블)를 1차 비교 기준으로 사용해야 한다. 코드 컨테이너(파일/모듈) 요약을 1차 비교 기준으로 사용하지 않는다.
- **FR-002**: 시스템은 코드 컨테이너/루틴 요약의 유사도가 낮다는 이유만으로 그 안의 Rule 후보를 전부 탈락시키지 않아야 한다(거친 단위가 후보를 0으로 만드는 하드 게이트가 되어선 안 된다). 컨테이너/루틴은 스코프·보조 신호로만 쓸 수 있다.
- **FR-003**: 매핑 동작은 분석 전략(framework/dbms)과 무관하게 **동일 경로**로 처리되어야 한다(전략별 분기 없이). dbms의 단위 복원은 기존 오너 복원 규칙을 재사용한다.
- **FR-004**: 부정확한 매핑 후보는 기존 검증·중재 단계가 걸러내야 한다(매핑 정밀도 유지). 본 피처는 후보 회수(recall)를 회복하되 정밀도 보호 장치를 약화시키지 않는다.
- **FR-005**: 본 피처는 생산자(analyzer) 그래프, 신규 Neo4j 라벨/관계, 프론트 표시를 변경하지 않아야 한다(소비자 매칭 로직만 정합).
- **FR-006**: 코드분석 풀런 후, **관련 코드가 존재하는 Task에는 1건 이상의 Rule 매핑이 생성**되어야 한다(현재의 전역 0건 상태를 재발시키지 않는다).
- **FR-007**: 규칙이 세션 범위로 많아질 때도 비교 비용이 비정상적으로 폭증하지 않도록 합리적 상한/우선순위를 두되, 매핑 품질을 떨어뜨리지 않아야 한다.

### Key Entities *(데이터 관련)*

- **BpmTask(업무 단계)**: 업무 프로세스의 한 단계. 매핑의 한쪽 끝.
- **Rule(코드 규칙)**: 분석된 코드에서 추출된 비즈니스 규칙. 본문 + 예시(Given/When/Then)를 가지며, 소속 루틴(함수/프로시저 등)·관련 테이블과 연결된다. 매핑의 다른 쪽 끝이자 **비교의 기본 단위**.
- **Task↔Rule 매핑**: "이 업무 단계는 이 코드 규칙으로 구현된다"는 관계. 본 피처가 0건 → 양수로 회복시키는 대상.
- **컨테이너/루틴(파일·클래스·모듈·함수·프로시저)**: 코드 구조 단위. 본 피처에서 **비교의 1차 기준이 아니라 보조 스코프**로 격하된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 코드분석 풀런에서, 코드가 구현하는 업무 단계의 매핑률이 **0%에서 의미 있는 수준으로 상승**한다(관련 Task 다수에 ≥1 규칙 매핑). 고립 규칙 비율이 크게 감소한다.
- **SC-002**: 파일당 단위가 많은 코드(예: 함수 17~21개/파일)와 거대 단위 코드(예: 다수 구문을 품은 프로시저) 모두에서 매핑이 생성된다(특정 코드 모양 의존 없음).
- **SC-003**: 매핑이 늘어도 **명백한 오매핑이 도배되지 않는다**(검증·중재가 부적합 후보를 계속 걸러낸다).
- **SC-004**: 동일 입력에 대한 후보 수가 직접 검증에서 **0 → N(>0)** 으로 회복된다(현 framework 그래프로 즉시 입증). DBMS 그래프 확보 시 동일 회복을 라이브로 확인한다.

## Assumptions

- 코드분석이 Rule(과 예시 Given/When/Then)을 충분히 구체적으로 추출해 둔다(현 analyzer 출력 기준). 비교의 구체성은 이 추출 품질에 의존한다.
- Rule은 한 인제스천 세션 범위(수십~약 100개)라, 컨테이너 게이트 없이 규칙을 직접 비교해도 비용·지연이 무시 가능하다(기존 컨테이너 게이트는 "수천 모듈" 대형 시스템 가정으로 도입되었다).
- 기존 매핑 저장·검증기·중재·전체탐색 SSE 인터페이스를 그대로 재사용한다(본 피처는 "무엇을 비교하는가"만 바꾼다).
- DBMS 라이브 검증은 DBMS 분석 그래프 확보 시 수행한다. 현재는 framework 그래프로만 검증 가능(미확보 구간은 "미검증"으로 명시).
- 생산자(analyzer)·프론트는 본 피처 범위 밖이며 변경하지 않는다.
