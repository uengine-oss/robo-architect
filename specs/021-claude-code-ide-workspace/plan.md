# Implementation Plan: Claude Code IDE Workspace

**Branch**: `021-claude-code-ide-workspace` | **Date**: 2026-05-09 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/021-claude-code-ide-workspace/spec.md`

## Summary

Convert the existing single-pane Claude Code panel (feature 015) into a 3-pane VS Code-style workspace: file tree (left), code editor (middle), Claude Code PTY terminal (right, unchanged behavior). The architect picks a working directory once, then browses, opens, edits, and saves files in the middle pane while Claude runs in the right pane. The tree refreshes manually for v1 (live FS watching deferred to v2). Editor reload-vs-keep prompt prevents Claude's writes from silently overwriting unsaved buffers.

**Technical approach**: Add three new REST endpoints under the existing `/api/claude-code` prefix (`tree`, `file` GET, `file` PUT) for filesystem operations scoped to a project root that the user has explicitly chosen. Reuse the existing terminal WebSocket and folder-picker endpoints unchanged. Frontend introduces a new `ClaudeCodeWorkspace.vue` shell that owns the 3-pane layout and embeds the existing `ClaudeCodeTerminal.vue` in the right pane verbatim. The middle pane uses CodeMirror 6 (chosen in research over Monaco for bundle-size reasons in a Vite SPA).

## Technical Context

**Language/Version**: Python 3.11+ (backend), Vue 3.5 + Vite 5 (frontend)
**Primary Dependencies**: FastAPI (existing), Pydantic (existing), CodeMirror 6 (`@codemirror/*` — new frontend dep), `@codemirror/view` + language packs for md/json/yaml/python/javascript/vue, xterm.js (existing, unchanged in right pane)
**Storage**: Local filesystem only — the chosen project directory; no Neo4j writes for this feature. The graph remains source of truth for *domain artifacts*, but the file tree pane is a filesystem view of generated outputs, not domain state.
**Testing**: Manual smoke per `quickstart.md` (8 scenarios). No automated test infrastructure for the existing Claude Code feature exists — this plan does not introduce one to keep scope bounded; constitutional follow-up captured in Complexity Tracking.
**Target Platform**: macOS / Linux (POSIX) — same constraint as feature 015. Browser is any evergreen (Chrome/Edge/Safari/Firefox). Single-user / self-hosted assumption inherited.
**Project Type**: Web application — `api/` (FastAPI) + `frontend/` (Vue SPA), per the existing repo layout.
**Performance Goals**: Tree initial render < 1s for ~5,000-file directories (root level only, lazy children). File open < 500ms for files ≤ 1MB. Save < 500ms for files ≤ 1MB. Terminal latency unchanged (~100ms localhost from feature 015).
**Constraints**: 2 MB hard editor file-size cap. Lazy load directory children (no full-tree walk on workspace open). Path-traversal sandboxing — every filesystem call scoped to the project root the user picked. POSIX-only host (Windows refused gracefully, same as terminal).
**Scale/Scope**: One workspace open at a time per browser tab (single-user assumption). Project trees up to a few tens of thousands of files. Editor optimized for files generated by RoboArchitect (PRDs, agent configs, specs) and by Claude (code) — typically < 200 KB.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| **I. Graph-as-Source-of-Truth** | ✅ PASS | Feature is a *filesystem* view; it does not store any domain state. Generated artifacts are still regenerable from Neo4j via the existing `setup-project` endpoint. The tree/editor never read or write Neo4j. |
| **II. Event Storming Vocabulary** | ✅ PASS | Feature speaks the filesystem (tree, file, directory) — no DDD vocabulary applies. No new graph nodes, so no naming conflict. |
| **III. Streaming-First UX** | ✅ PASS (with note) | Tree/file ops are millisecond-scale → request/response is correct per the constitution's carve-out ("pure request/response is reserved for instant graph queries", which by analogy covers instant filesystem queries). The terminal pane continues to stream over WebSocket. |
| **IV. Human-in-the-Loop on Mutations** | ✅ PASS | All editor saves are explicit user actions (Save shortcut / button). External-modification reload prompt (FR-011) is itself a human-in-the-loop checkpoint protecting the user's unsaved edits from Claude's writes. |
| **V. Feature-Modular Architecture** | ✅ PASS | New code lives under existing `api/features/claude_code/` and `frontend/src/features/claudeCode/` — no new feature module, no cross-feature imports. The right pane embeds the existing terminal component without modification. |
| **VI. Provider-Agnostic LLM Runtime** | ✅ PASS | No LLM calls in this feature. Claude is invoked via the existing PTY in the right pane, which is already provider-agnostic at the CLI boundary. |
| **VII. Observable by Default** | ⚠️ PARTIAL | Filesystem ops are simple enough that per-request correlation IDs are overkill. We will log structured warnings on the only multi-step operation (path-resolution sandbox violations) since those are the only "fail in non-obvious ways across boundaries" risk. Captured in research D5. |

**Gate result**: PASS. The Observable-by-default partial is justified because filesystem reads/writes are single-step synchronous operations that fail with their own clear errors; SmartLogger + correlation IDs are designed for multi-stage LLM/network/graph pipelines and would be ceremonial here.

**Structure Decision**: Reuse the existing `claude_code` feature module on both backend and frontend (Constitution principle V). All new files live under those existing folders; no new feature directory is created. The existing folder picker (`/browse-directory`) and the existing terminal WebSocket (`/terminal`) endpoints are reused **unchanged** — the workspace is an additive layer on top, not a replacement of the underlying transport. `App.vue` swaps which Vue component renders for the `Claude Code` tab; the swap is the only change to the global layout.
