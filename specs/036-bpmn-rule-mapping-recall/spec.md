## 배경 및 문제 *(context)*

요구사항 탭에서 BPMN 모델을 탐색하면, 레거시 코드 분석 그래프의 비즈니스 로직(BL) 룰들이 각 BPMN 활동(Task)에 자동으로 매핑된다. 매핑은 후보 검색(retrieval) → LLM 검증(validator) → 화면 표시(accept된 매핑 + 소수의 near-miss 후보)의 단계로 진행되며, 사용자는 최종 결과만 검토한다.

문제는 **어휘 분포의 불일치**다. 사용자가 작성한 요구사항/활동 설명은 일상적 평문(예: "정기 출금을 처리한다")인 반면, 레거시 코드에서 추출된 룰은 개발자 용어(약어·테이블명·함수명, 예: `autoPay`, `TB_WD_MST`, `procWithdraw`)로 표현된다. 두 표현은 의미가 같아도 텍스트 유사도가 낮게 나오고, 그 결과 **실제로는 매핑되어야 할 룰이 검증 단계에 도달하기도 전에 후보 검색에서 탈락**한다(recall 손실). 사용자는 "분명히 이 활동에 해당하는 규칙이 코드에 있는데 매핑이 안 떴다"는 누락을 경험한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 어휘가 다른데 의미가 같은 룰이 매핑됨 (Priority: P1)

분석가가 BPMN 활동을 탐색할 때, 활동 설명은 평문이고 대응하는 레거시 코드 룰은 개발자 약어로 쓰여 있다. 시스템은 두 표현 사이의 용어 차이를 메워, 의미상 대응하는 룰을 활동에 매핑해 보여준다.

**Why this priority**: 이 기능의 존재 이유 그 자체다. 어휘갭으로 인한 누락이 핵심 통증이며, 이걸 해결하지 못하면 나머지는 의미가 없다.

**Independent Test**: 평문 활동명과 개발자 약어로 표현된 룰이 의미상 1:1 대응하도록 구성한 데이터셋에서, 개선 전에는 매핑되지 않던 룰이 개선 후 매핑 결과에 나타나는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 활동 "정기 출금을 처리한다"와, 같은 의미를 약어로 표현한 레거시 룰(`autoPay`/`TB_WD_MST` 기반)이 존재하고 개선 전에는 후보에서 탈락하던 상태일 때, **When** 매핑을 실행하면, **Then** 해당 룰이 이 활동의 매핑 결과(accept)로 나타난다.
2. **Given** 동일한 데이터셋, **When** 개선 전/후의 매핑 결과를 비교하면, **Then** 어휘갭으로 누락되던 정당한 룰의 매핑 수가 증가한다(recall 향상).

---

### User Story 2 - 검토할 화면 항목이 늘어나지 않음 (Priority: P1)

분석가가 매핑 결과를 검토할 때, recall이 올라갔다고 해서 화면에 표시되는 항목(매핑된 룰 + 검토 대상 near-miss)이 이전보다 늘어나 인지부하가 커지면 안 된다. 늘어난 후보는 내부 검증 단계가 흡수하고, 사용자는 여전히 정제된 결과만 본다.

**Why this priority**: 사용자가 명시한 **최우선 목표**다. recall을 올리되 사용자 화면을 그대로 유지하는 것이 이 스펙의 성패 기준이다. P1으로 US1과 동등하게 둔다.

**Independent Test**: 같은 입력에 대해 개선 전/후 사용자에게 표시되는 항목 수(accept 매핑 수 + near-miss reject 표시 수)를 집계해, 증가하지 않음을 확인한다.

**Acceptance Scenarios**:

1. **Given** 임의의 BPMN 문서와 레거시 코드 그래프, **When** 개선 후 매핑을 실행하면, **Then** 활동당 사용자에게 노출되는 near-miss 후보 수는 개선 전과 동일한 상한 이내로 유지된다.
2. **Given** recall이 향상되어 더 많은 룰이 내부적으로 후보가 된 경우, **When** 결과를 표시하면, **Then** 추가된 후보는 검증 단계에서 accept 또는 폐기로 처리되어 사용자 화면의 검토 부담을 늘리지 않는다.

---

### User Story 3 - 처리 비용이 상한 안에서 유지됨 (Priority: P2)

운영자/분석가 관점에서, recall 개선이 검증 단계의 처리량 폭증으로 이어져 매핑 소요 시간이나 비용이 통제 불능이 되면 안 된다. 후보 확장은 용어 정규화로 신호 품질을 높이는 방식이어야 하며, 검증 후보 예산은 기존 상한을 유지한다.

**Why this priority**: 인지부하(화면)는 P1에서 이미 고정. 비용은 사용자가 "상한 유지"로 못박은 제약이라 중요하지만, 화면 누락 해결보다는 후순위.

**Independent Test**: 동일 입력에 대해 개선 전/후 검증 단계가 판정하는 활동당 후보 수와 매핑 총 소요 시간을 측정해, 사전 정의된 상한을 넘지 않음을 확인한다.

**Acceptance Scenarios**:

1. **Given** 대형 레거시 모듈(활동당 수십~수백 룰), **When** 개선 후 매핑을 실행하면, **Then** 검증 단계가 활동당 판정하는 후보 수는 기존 상한 이내로 유지된다.
2. **Given** 동일 입력, **When** 매핑 전체 소요 시간을 측정하면, **Then** 개선 전 대비 사전 정의된 허용 배수 이내로 유지된다.

---

### Edge Cases

- **용어 사전이 비어 있거나 추출 실패**: 정규화에 쓸 용어 매핑이 없으면 시스템은 기존(정규화 없는) 매핑 동작으로 그대로 진행하며 오류 없이 결과를 낸다.
- **잘못된/과도한 정규화**: 용어 매핑이 부정확해 무관한 룰의 유사도가 올라가면 검증 단계가 이를 폐기해야 하며, 그 결과 사용자 화면에 노이즈가 새어 나가지 않아야 한다.
- **동음이의 용어**: 같은 약어가 서로 다른 업무 범주에서 다른 의미로 쓰일 때, 정규화가 한 의미로 과도하게 수렴시켜 다른 범주의 정당한 매핑을 가리지 않아야 한다.
- **정규화로도 메워지지 않는 갭**: 사전에 없는 신규/특이 용어로 인한 누락은 이 스펙의 범위가 아니며(후속), 회귀(기존에 되던 매핑이 깨짐)만 발생시키지 않으면 된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 BPMN 활동과 레거시 BL 룰을 매핑하는 후보 검색 단계에서, 두 텍스트의 평문↔개발자 용어 차이를 메우는 **용어 정규화**를 적용해야 한다.
- **FR-002**: 시스템은 용어 정규화에 사용할 용어 대응 정보를, 이미 도메인에서 추출되는 용어 사전(glossary)을 재사용하여 확보해야 한다(별도 신규 입력 요구 없음).
- **FR-003**: 용어 정규화로 인해 후보 검색을 통과하는 룰이 늘어나더라도, **사용자에게 노출되는 항목(매핑된 룰 + near-miss 후보)의 수는 개선 전 상한을 초과하지 않아야** 한다.
- **FR-004**: 늘어난 후보는 기존 검증 단계가 accept/폐기로 판정해야 하며, 폐기된 후보는 사용자 화면에 표시되지 않아야 한다.
- **FR-005**: 시스템은 검증 단계가 활동당 판정하는 후보 수 상한과, 후보 검색 통과 기준(유사도 floor)을 개선 전과 동일하게 유지해야 한다. 즉 recall 향상은 통과 기준을 낮추거나 후보 예산을 늘리는 방식이 아니라, 정규화로 신호 품질을 높이는 방식으로만 달성한다. 이에 따라 동일 입력의 매핑 전체 소요 시간은 개선 전 대비 **1.2배를 넘지 않아야** 한다.
- **FR-006**: 용어 사전이 없거나 정규화를 적용할 수 없는 경우, 시스템은 오류 없이 기존(정규화 미적용) 동작으로 매핑을 완료해야 한다.
- **FR-007**: 시스템은 이번 개선이 매핑 결과 그래프에 새로운 노드 종류나 관계 종류를 추가하지 않아야 한다(기존 매핑 산출물 형태 유지).
- **FR-008**: 시스템은 개선 효과를 측정할 수 있도록, 정규화 적용 여부에 따른 매핑 결과 차이(새로 매핑된 룰, 사용자 노출 항목 수)를 비교 가능한 형태로 확인할 수 있어야 한다.

### Key Entities *(include if feature involves data)*

- **BPMN 활동(Task)**: 사용자가 탐색하는 프로세스 단위 작업. 평문 이름/설명과 수행 주체(actor)를 가진다. 룰 매핑의 대상.
- **BL 룰(Rule)**: 레거시 코드 분석에서 추출된 비즈니스 로직 단위. 개발자 용어(함수명·테이블명·약어)와 GIVEN/WHEN/THEN 형태의 조건을 가진다. 활동에 매핑되는 후보.
- **용어 사전 항목(Glossary Term)**: 도메인 용어와 그 의미/동의어/대응 개발자 표현의 매핑. 정규화의 원천.
- **활동-룰 매핑(Activity–Rule Mapping)**: 검증을 통과한 활동↔룰 연결. 이 스펙의 최종 산출물이며 형태는 기존과 동일.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 골든 평가셋(자동납부 본인확인 업무흐름 PDF 2종 + 레거시 C코드 분석 그래프)에서, 개선 전 어휘갭으로 누락되던 정당한 룰이 **최소 1건 이상 매핑으로 회복**되며, **개선 전에 이미 매핑되던 결과는 단 1건도 깨지지 않는다(회귀 0건)**.
- **SC-002**: 사용자의 **검토 부담(노이즈)**이 증가하지 않는다 — 구체적으로 활동당 노출되는 **near-miss reject 후보 수가 개선 전 상한(구조적 cap)을 초과하지 않는다.** (정정: 구현 측정 결과, 회복된 정확한 매핑(accept)은 recall 향상의 산출물로서 증가하는 것이 정상이며 이는 노이즈가 아니다. 따라서 "표시 항목 수 불변"은 accept가 아니라 검토 부담인 near-miss에 적용한다. task당 회복 수는 회복 상한 노브로 조절한다.)
- **SC-003**: 검증 단계가 활동당 판정하는 후보 수는 개선 전 상한 이내로 유지되고, 동일 입력의 전체 매핑 소요 시간은 개선 전 대비 **1.2배 이내**로 유지된다.
- **SC-004**: 용어 사전이 비어 있는 경우에도 매핑이 오류 없이 완료되며, 기존에 매핑되던 결과가 깨지지 않는다(회귀 0건).
- **SC-005**: 잘못된 정규화로 무관한 룰이 후보에 진입하더라도, 그것이 사용자 화면의 매핑 또는 near-miss 표시로 새어 나가지 않는다.

## Assumptions

- **인지부하의 정의**: 사용자 인지부하는 "사용자가 검토해야 하는 화면 항목 수(accept된 매핑 + 표시되는 near-miss 후보)"로 측정한다. 이 수치를 늘리지 않는 것이 최우선 제약이다.
- **검증 단계 = 완충재**: 최종 accept/폐기를 판정하는 LLM 검증 단계가 이미 존재하며, 후보가 늘어도 사용자 노출을 막는 역할을 한다고 가정한다(기존 파이프라인 재사용).
- **용어 사전 재사용**: 도메인 용어 사전이 이미 추출되고 있으며(별도 신규 수집 불필요), 이를 정규화 원천으로 재사용한다.
- **범위 한정 (v1)**: 이번 스펙은 **용어 정규화**만 다룬다. 다중신호 검색(어휘/구조 신호 결합), 적응형 통과 기준(분포 기반 동적 floor)은 명시적으로 범위 외이며 후속 스펙으로 미룬다.
- **비용 상한 유지**: 후보 검색 통과 기준과 검증 후보 예산은 개선 전 값을 유지한다. recall은 통과 기준 완화가 아니라 신호 품질 향상으로만 얻는다.
- **그래프 스키마 불변**: 신규 노드 라벨/관계는 추가하지 않으며, 기존 매핑 산출물 형태를 그대로 사용한다(기존 자산 재배선 중심).
