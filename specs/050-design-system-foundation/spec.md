# Feature Specification: 디자인 시스템 토대 (Design System Foundation)

**Feature Branch**: `050-design-system-foundation` (미생성 — 사용자 WIP 보존을 위해 착수 승인 시 분기)

**Created**: 2026-07-15

**Status**: Draft

**Input**: 사용자 요청 — "기능은 두고 표면(레이아웃·시각 스타일·컴포넌트)을 다시 디자인하라. 지금 스타일이 조잡하다. 범위는 Robo Architect 영역 전 탭(Legacy=Analyzer 제외), 톤은 알아서 세련되게." + UI 견적(오토모드·Ctrl+C·경로D·로그 스크롤·tool 표기·모드 설명·세션매니저 500 등)에서 도출된 표면 결함.

## 배경과 문제 (Why)

Robo Architect의 각 화면은 공용 규칙 없이 제각각 만들어져 있다. 확인된 근거:

- **스트림 로그가 5벌로 복붙**돼 있고(`ProposalCreate`·`ProposalDetail`·`PlanView`·`OdaStandardTrack`·`SandboxProgressView`), 자동 스크롤은 `ProposalCreate` 1벌만 가지고 있다.
- **툴 활동이 원시 문자열**로 노출된다: `[tool] Read d:\work\robo\...\output-schema.md`.
- **오버레이 닫힘 규칙이 제각각**이다 — 폴더 피커/세션 매니저는 바깥 클릭(`@click.self`)으로 닫혀 "커서가 벗어나면 꺼진다"는 체감을 만든다.
- 상태 배지·도메인 색·버튼·간격에 단일 토큰이 없어 화면마다 미세하게 다르다.

이것은 "스타일이 안 예쁨"이 아니라 **공용 표현 계층(디자인 토큰 + 공용 컴포넌트)이 없다**는 구조 문제다. 이 스펙은 그 단일 표현 계층을 만들어(Constitution 원칙 III — 단일 진실), 이후 탭별 표면 적용(051+)과 기능 결함 수정(별도 스펙)이 그 위에 얹히게 한다.

## 범위 경계 (Scope)

**이 스펙(050) = 표면 토대만.** 기능·로직·데이터 흐름·화면 배선은 **동결**한다.

- 포함: 디자인 토큰(색/간격/타이포/상태색/도메인색), 공용 UI 컴포넌트 키트, 그리고 그 컴포넌트에 내장되는 3개 상호작용 계약(로그 자동스크롤·사람 친화 표기, 오버레이 닫힘 정책, 상태/도메인 색의 단일 매핑).
- 제외(별도 스펙):
  - 탭별 시각 적용(051 Proposals, 052 Code, 053 Stories/Process/Design/Data …).
  - **기능/동작 수정**(상시 오토모드, Windows Ctrl+C, 경로 D, 세션매니저 500, 폴더피커 드라이브 전환) — 표면이 아니라 동작 변경이므로 회귀 테스트를 동반한 **별도 behavioral 스펙**으로 분리.
  - Legacy 탭(= Analyzer Module Federation 영역).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 하나의 시각 언어 (Priority: P1)

Architect의 어느 탭을 열어도 같은 색·간격·컴포넌트·타이포로 보여, 사용자가 "같은 제품"이라고 느낀다. 개발자는 새 화면을 만들 때 색·간격을 즉흥적으로 정하지 않고 토큰과 공용 컴포넌트를 가져다 쓴다.

**Why this priority**: "조잡함"의 근본 원인 제거. 이 토대가 없으면 이후 모든 탭 재설계가 다시 제각각이 된다.

**Independent Test**: 토큰과 공용 컴포넌트 키트를 만든 뒤, 대표 화면 1개(Proposals)를 키트로 렌더해 하드코딩 색/간격이 0인지, 다른 화면에서 같은 컴포넌트가 동일하게 보이는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 토큰 세트가 정의된 상태, **When** 임의의 마이그레이션된 화면을 검사하면, **Then** 색·간격·상태색·도메인색이 모두 토큰을 통해서만 지정되고 인라인 hex/매직픽셀이 없다.
2. **Given** 상태 배지가 여러 화면에 있을 때, **When** 상태 색 매핑을 한 곳에서 바꾸면, **Then** 모든 화면의 같은 상태 배지가 함께 바뀐다.
3. **Given** 강조색 토큰을 한 값 바꾸면, **When** 앱을 다시 렌더하면, **Then** 버튼·활성 탭·포커스 링 등 강조가 쓰인 모든 곳이 일괄 반영된다.

---

### User Story 2 - 읽을 수 있는 활동 로그 (Priority: P1)

AI 분석/구현이 진행될 때, 로그가 사람이 읽을 수 있는 행으로 보이고 항상 최신 줄로 자동 스크롤된다. 사용자는 로그를 손으로 끝까지 내리지 않아도 현재 상황을 본다.

**Why this priority**: 견적 #2(tool 친화)·#5(자동 스크롤)의 직접 해소. 사용자가 가장 자주 보는 진행 화면의 체감을 좌우한다.

**Independent Test**: 공용 스트림 로그 컴포넌트에 툴 이벤트를 흘려보내, 원시 `[tool] Read <경로>`가 아니라 "참조 문서 읽는 중 + 파일명" 형태로 렌더되고, 새 줄이 추가될 때 하단으로 자동 스크롤되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 분석 스트림이 진행 중, **When** 새 로그 줄이 추가되면, **Then** 로그 뷰가 자동으로 최신 줄까지 스크롤된다(모든 스트림 뷰에서 동일).
2. **Given** 툴 사용 이벤트(Read/Write/MCP 질의/사고/완료), **When** 로그에 표시되면, **Then** 아이콘 + 사람이 읽는 라벨 + 파일/대상 칩으로 표시되고 종류별로 색이 구분된다.
3. **Given** 사용자가 과거 로그를 보려고 위로 스크롤한 상태, **When** 새 줄이 도착하면, **Then** 강제로 하단으로 끌어내리지 않고 사용자의 스크롤 위치를 존중한다(하단 근처일 때만 자동 추종).

---

### User Story 3 - 실수로 닫히지 않는 오버레이 (Priority: P2)

모달·팝오버·폴더 피커는 사용자가 명시적으로 닫을 때(×, 취소, Esc)만 닫힌다. 마우스가 바깥으로 나가거나 옆을 스치는 클릭으로는 닫히지 않는다.

**Why this priority**: 견적 #1/#9("커서 벗어나면 팅긴다") 해소. 입력 중이던 내용을 잃는 좌절을 막는다.

**Independent Test**: 공용 오버레이 컴포넌트를 띄우고, 바깥 영역으로 마우스를 이동/클릭했을 때 닫히지 않으며 ×·취소·Esc로만 닫히는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 공용 오버레이가 열려 있고 사용자가 안에 입력 중, **When** 마우스가 오버레이 밖으로 나가거나 바깥을 클릭하면, **Then** 오버레이는 닫히지 않고 입력이 보존된다.
2. **Given** 오버레이가 열려 있을 때, **When** 사용자가 ×/취소/Esc를 누르면, **Then** 오버레이가 닫힌다.

---

### User Story 4 - 기능 불변 보장 (Priority: P1)

표면 토대를 도입해도 어떤 버튼의 동작, 데이터, 화면 배선도 바뀌지 않는다.

**Why this priority**: 사용자의 핵심 제약("기능은 두고"). 회귀는 이 스펙의 실패다.

**Independent Test**: 마이그레이션된 화면의 기존 상호작용을 실제로 조작해, 클릭 동작·API 호출·상태 변화가 이전과 동일한지 확인한다(기존 컴포넌트 테스트 무회귀).

**Acceptance Scenarios**:

1. **Given** 어떤 화면을 공용 키트로 교체, **When** 기존 버튼/입력을 조작하면, **Then** 이전과 동일한 동작·이벤트·API 호출이 발생한다(순수 표현 교체).
2. **Given** 미마이그레이션 화면이 남아 있을 때, **When** 앱을 사용하면, **Then** 토큰/키트 도입이 미마이그레이션 화면을 깨뜨리지 않는다(점진 도입 가능).

### Edge Cases

- 스트림 로그에 매우 긴 경로/줄이 올 때 가로 스크롤이 페이지 본문을 밀지 않고 로그 내부에서만 처리되는가?
- 로그가 폭주(초당 수십 줄)할 때 자동 스크롤이 성능 저하 없이 동작하는가?
- 사용자가 로그 중간을 선택(드래그)해 복사 중일 때 자동 스크롤이 선택을 방해하지 않는가?
- Esc가 여러 오버레이가 겹친 상태에서 최상위 하나만 닫는가?
- 폭이 좁아질 때(반응형) 좌측 레일·세션 탭·모드 카드가 레이아웃을 밀지 않고 접히거나 스크롤되는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 색·간격·타이포·반경·상태색·도메인 요소색을 **단일 토큰 세트**로 정의하고, 모든 표현이 그 토큰을 통해서만 색/간격을 지정해야 한다(인라인 hex·매직픽셀 금지). *(Constitution III)*
- **FR-002**: 도메인 요소 색(UserStory/Aggregate/Command/Event/Policy/ReadModel/UI)과 Proposal 상태 색(Intent/Plan/구현중/검증중/승인대기/완료/폐기)은 **한 곳의 의미 매핑**에서 나와, 카드·배지·범례·상세패널 어디서나 동일해야 한다.
- **FR-003**: 시스템은 다음 **공용 UI 컴포넌트**를 각 1벌로 제공해야 한다: 스트림 로그(StreamLog), 오버레이/모달(AppModal), 상태 배지(StatusPill), 도메인 칩(DomainChip), 버튼(Button), 분해모드 카드(ModeCard), 터미널 크롬(TerminalChrome), 세션 탭(SessionTabs).
- **FR-004**: StreamLog는 새 줄 도착 시 **하단 자동 스크롤**을 기본 제공하되, 사용자가 위로 스크롤한 경우 위치를 존중하고(하단 근처일 때만 추종) 텍스트 선택을 방해하지 않아야 한다.
- **FR-005**: StreamLog는 툴 활동 줄을 **사람 친화 표기**(아이콘 + 라벨 + 대상 칩)로 렌더하고 종류(읽기/쓰기/MCP 질의/사고/완료/오류)를 색으로 구분해야 한다. 원시 `[tool] …` 문자열을 그대로 노출하지 않는다.
- **FR-006**: AppModal은 **명시적 의도(× 버튼·취소·Esc)로만 닫히며**, 마우스 이탈이나 우발적 바깥 클릭으로 닫히지 않아야 한다.
- **FR-007**: 공용 컴포넌트는 **표현만 담당**하고, 어떤 데이터 취득·상태 변경·이벤트 배선도 포함하지 않아야 한다(입력은 props, 동작은 상위가 주입). 도입은 화면 단위로 **점진 가능**해야 하며 미마이그레이션 화면을 깨뜨리지 않는다.
- **FR-008**: 시스템은 **의도적 다크 단일 세계**로 표현을 확정한다(제품 정체성). 향후 라이트 확장이 가능하도록 색은 토큰으로 분리하되, 이 스펙의 산출물은 다크를 기준으로 한다.
- **FR-009**: 상호작용 요소는 **키보드 포커스 표시**를 가지며, 애니메이션은 `prefers-reduced-motion`을 존중해야 한다.
- **FR-010**: 넓은 콘텐츠(로그·경로·표)는 **자체 컨테이너에서만 가로 스크롤**되고 페이지 본문을 가로로 밀지 않아야 한다. 행 정렬은 grid/gap·상대 크기로 고정해 지그재그를 만들지 않는다. *(원칙 7)*
- **FR-011**: 표면 교체는 **기능 회귀 0**이어야 한다 — 기존 동작·이벤트·API 호출·데이터가 불변임을 회귀로 확인한다.

### Key Entities

- **디자인 토큰**: 색(ground/surface/elevated/line/text/dim/accent), 상태색, 도메인색, 간격 스케일, 타이포 스케일, 반경 — 표현의 단일 진실.
- **의미 색 매핑**: (도메인 요소 → 색), (Proposal 상태 → 색·라벨) 두 매핑을 한 곳에서 정의.
- **공용 컴포넌트 키트**: StreamLog, AppModal, StatusPill, DomainChip, Button, ModeCard, TerminalChrome, SessionTabs — props로만 구동되는 순수 표현 부품.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 마이그레이션된 화면에서 인라인 색/매직픽셀 사용 건수 = **0** (전부 토큰 경유).
- **SC-002**: 스트림 로그 구현이 **5벌 → 1벌**로 수렴하고, 모든 스트림 뷰에서 자동 스크롤이 동작한다.
- **SC-003**: 로그의 툴 활동 줄에서 원시 `[tool] … <풀경로>` 노출 = **0** (전부 사람 친화 표기).
- **SC-004**: 마우스 이탈·우발적 바깥 클릭으로 닫히는 오버레이 = **0** (× / 취소 / Esc로만 닫힘).
- **SC-005**: 상태·도메인 색 매핑을 한 곳에서 바꾸면 앱 전체의 해당 표식이 함께 바뀐다(단일 진실 검증).
- **SC-006**: 표면 교체로 인한 기능 회귀 = **0** (기존 상호작용 테스트 무회귀 + 실물 조작 확인).

## Assumptions

- 다크 단일 세계는 의도적 선택이다(제품 정체성). 라이트 테마는 후속 확장 여지로만 둔다.
- Legacy 탭(Analyzer MF 영역)은 이 작업 범위 밖이다.
- Proposals·Code 탭을 대표 검증 화면으로 삼는다(사용자 실사용 화면).
- 기존 Vue 스토어·API·SSE/WebSocket 배선은 **그대로 재사용**한다(이 스펙은 표현 계층만).
- 기능/동작 결함(상시 오토모드·Windows Ctrl+C·경로 D·세션매니저 500·폴더피커 드라이브 전환)은 이 스펙이 아니라 **별도 behavioral 스펙**에서 회귀 테스트와 함께 처리한다.
- 현재 main 작업 트리에 사용자 WIP(`skill_runner.py`·`ClaudeCodeTerminal.vue` 등)가 있으므로, 구현 착수 시 분기·WIP 처리를 먼저 사용자와 합의한다.
