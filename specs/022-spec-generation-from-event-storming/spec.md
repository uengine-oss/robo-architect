# Feature Specification: DDD Artifact Generation from Event Storming

**Feature Branch**: `022-spec-generation-from-event-storming`
**Created**: 2026-05-09
**Last Updated**: 2026-05-12 (added frontend stories P5–P7; reconciled spec with shipped code paths)
**Status**: Draft
**Input**: User description: "이벤트 스토밍 그래프(Neo4j)로부터 spec.md/plan.md 같은 SpecKit 산출물 대신, 'DDD for SDD' 글의 DDD 아티팩트 형식(domain-terms / bc-canvas / context-map / aggregate-spec / acl)을 자동 생성한다. Figma 토큰은 필요 없다 — 와이어프레임 씬그래프가 이미 우리 Neo4j 안에 있으므로 거기서 만든다."

**2026-05-12 update**: Extend the feature to own the *consumer-side packaging* that the PRD-generation flow now emits alongside the DDD artifact set — Claude/Cursor skills, slash commands, and a CLAUDE.md/PRD.md split — and add a **frontend perspective**: declare the target frontend framework up-front in the PRD UI, project the event-modeled UI flow into a system-wide menu/navigation artifact under `specs/frontend/` (a sibling of `specs/bounded-contexts/`, *not* a child of any BC folder), drop per-BC agent declarations in favour of role-based agents (Frontend Engineer, DDD Specialist) that reference the existing skills, and add a `/generate-frontend` slash command.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Generate the full DDD artifact set for one Bounded Context (Priority: P1)

A domain modeler has finished event-storming a Bounded Context — its Aggregates with data models, Commands with Given/When/Then criteria, Events, Policies, ReadModels, User Stories, and UI wireframes (open-pencil scene graphs synced from the Figma plugin). They trigger "Generate DDD Artifacts" on that Bounded Context. The system writes, under `specs/bounded-contexts/<bc-slug>/`:

- `domain-terms.md` — the Ubiquitous Language glossary (one `## Term:` entry per Aggregate, Command, Event, ReadModel, and key Property in the BC, each with Definition / Business Context / Related Terms / Aliases to AVOID).
- `bc-<bc-slug>.md` — the Bounded Context Canvas (Purpose, Strategic Classification, Inbound/Outbound Communication tables, Ubiquitous Language summary, Business Decisions, Assumptions).
- `aggregates/aggregate-<aggregate-slug>.md` — one Aggregate Design Spec per Aggregate (Description, Aggregate Root, Member Entities & Value Objects, Properties table with Mutability, Enforced Invariants in EARS notation, Corrective Policies, Commands table, Domain Events Emitted, Repository Interface).
- `acl-<external-system-slug>.md` — one Anti-Corruption Layer Spec per external-system integration the BC has (Purpose, Boundary, Translation Maps, Error Translation, Idempotency, Forbidden Concepts) — only if the graph models such integrations.
- `requirements.md` — the BC's User Stories with Acceptance Criteria rendered in EARS notation, grouped by Aggregate, with each story's UI wireframe rendered from the stored scene graph (textual element tree always; an SVG image when the open-pencil service can render it) and the raw scene-graph JSON attached for fidelity.

The modeler reviews the folder and uses it as the spec layer their downstream AI coding agents consume.

**Why this priority**: This is the core value of the feature. The "DDD for SDD" thesis is that AI coding agents need these artifacts as a deterministic semantic contract; the differentiator here is that we are not asking an LLM to *invent* the aggregate design — we are *transcribing* a structured, human-authored event-storming graph into the DDD artifact format. A single-BC generator is the smallest viable unit and is independently shippable.

**Independent Test**: Pick one fully-modeled Bounded Context from the existing graph, invoke single-BC generation, and verify (a) the folder is created at `specs/bounded-contexts/<bc-slug>/` with `domain-terms.md`, `bc-<slug>.md`, one `aggregates/aggregate-<slug>.md` per Aggregate, and `requirements.md`; (b) every Aggregate, Command, Event, ReadModel, and User Story in the graph for that BC appears in the corresponding artifact; (c) Given/When/Then criteria appear as EARS `WHEN…/IF…/THEN system SHALL…` lines in both the Aggregate Spec's Enforced Invariants and `requirements.md`'s Acceptance Criteria; (d) each User Story's wireframe scene graph is reflected in `requirements.md`. No other feature needs to exist.

**Acceptance Scenarios**:

1. **Given** a Bounded Context with ≥1 Aggregate (each with attributes and ≥1 Command that has GWT) and ≥1 User Story, **When** the modeler triggers DDD-artifact generation for that BC, **Then** the folder `specs/bounded-contexts/<bc-slug>/` exists with `domain-terms.md`, `bc-<bc-slug>.md`, `aggregates/aggregate-<slug>.md` (one per Aggregate), and `requirements.md`, all populated from the graph.
2. **Given** an Aggregate whose Command has GWT criteria, **When** generation runs, **Then** the Aggregate Spec's "Enforced Invariants" section contains a numbered EARS line per GWT criterion (`WHEN <when> IF <given> THEN system SHALL <then>`), and the same criterion appears under that Command's User Story in `requirements.md`.
3. **Given** an Aggregate with attributes that carry mutability hints in the graph (e.g. "immutable after creation"), **When** generation runs, **Then** the Aggregate Spec's Properties table reproduces each attribute with its Type and Mutability.
4. **Given** a User Story bound to a UI node that has a stored scene graph, **When** generation runs, **Then** `requirements.md` includes that story's UI as a textual element tree (containers, text labels, buttons, inputs) and the raw `sceneGraph` JSON is written alongside (e.g. `requirements.assets/<userStoryId>-<ui-slug>.scene.json`).
5. **Given** the modeler regenerates against an unchanged graph with overwrite, **When** generation runs, **Then** every generated `.md` file is byte-identical to the previous run (excluding any `Generated:` timestamp line).
6. **Given** a Bounded Context with no Aggregates and no User Stories, **When** generation is triggered, **Then** the operation refuses with a clear validation message and creates no folder.

### User Story 2 — Regenerate the system-wide Context Map (Priority: P2)

After modeling or revising several Bounded Contexts and the Policy/Event flows between them, an architect wants the single system-level `specs/context-map.md` regenerated: a Mermaid `graph LR` of all BCs with their relationships, plus a Relationships section that names each pair's pattern (Customer-Supplier / Conformist / ACL / Open Host Service / Published Language / Shared Kernel / Separate Ways), a justification, and a pointer to the spec file that enforces it.

**Why this priority**: The Context Map is the artifact the article calls "the most important for AI" after the Aggregate Spec — it tells the agent where to put ACLs and where to type through shared schemas. It is sequenced second because it is system-wide and only meaningful once at least one BC's artifacts exist.

**Independent Test**: With ≥2 Bounded Contexts in the graph that have at least one cross-BC Policy/Event flow, invoke context-map generation and verify `specs/context-map.md` contains a Mermaid diagram listing every BC, a Relationships entry per cross-BC edge with a pattern label and a spec-file pointer, and that the diagram is syntactically valid Mermaid.

**Acceptance Scenarios**:

1. **Given** three Bounded Contexts with cross-BC Policy→Event flows between two of the pairs, **When** the architect triggers context-map generation, **Then** `specs/context-map.md` exists with a Mermaid `graph LR` showing all three BCs and the two relationship edges, and a Relationships section with one `### A → B` block per edge.
2. **Given** the graph does not record an explicit relationship pattern for an edge, **When** generation runs, **Then** the edge is rendered with a heuristic best-guess pattern (default: Customer-Supplier), the `### A → B` block notes "(inferred — confirm)", and a warning is reported so a human reviews it.
3. **Given** the graph has only one Bounded Context, **When** generation runs, **Then** `specs/context-map.md` is still produced with that single BC and an empty Relationships section, and a note explains there are no cross-BC relationships yet.

### User Story 3 — Refresh a single Aggregate Design Spec (Priority: P3)

After editing one Aggregate's invariants or commands in the event-storming UI, the modeler wants just that Aggregate's `aggregate-<slug>.md` regenerated, without rewriting the whole BC folder.

**Why this priority**: Quality-of-life for the tight iteration loop on a single Aggregate. Independently useful but depends on P1's per-Aggregate rendering existing.

**Independent Test**: Generate a BC folder, modify one Aggregate in the graph, invoke single-Aggregate refresh, and verify only that `aggregates/aggregate-<slug>.md` changed (mtime advanced) and it reflects the new graph state; sibling files in the BC folder are untouched.

**Acceptance Scenarios**:

1. **Given** a BC folder already exists, **When** the modeler triggers single-Aggregate refresh for one Aggregate, **Then** only `specs/bounded-contexts/<bc-slug>/aggregates/aggregate-<slug>.md` is rewritten; `domain-terms.md`, `bc-<slug>.md`, `requirements.md`, and the other aggregate specs are untouched.
2. **Given** the Aggregate to refresh does not yet have a spec file, **When** refresh runs, **Then** the file is created (and the parent `aggregates/` directory if needed) without disturbing siblings.

### User Story 4 — Bootstrap DDD artifacts for the whole model (Priority: P4)

An architect wants to generate, in one shot and with progress feedback, the Context Map plus every Bounded Context's full artifact folder for the entire model.

**Why this priority**: The productive "stand up the whole spec layer" workflow once the per-BC and context-map generators are trusted. Strictly composed of P1 + P2 and sequenced last.

**Independent Test**: Invoke whole-model generation against a graph with N Bounded Contexts and verify a streamed progress trace, N BC folders, and one `specs/context-map.md`, with a final summary listing everything created/skipped.

**Acceptance Scenarios**:

1. **Given** a graph with three Bounded Contexts, **When** the architect triggers whole-model generation, **Then** progress is streamed (one event per BC and per phase), three BC folders and one `context-map.md` are produced, and the final summary lists all four artifacts.
2. **Given** one BC already has a folder and overwrite is not requested, **When** whole-model generation runs, **Then** that BC is skipped and reported, the others and the context map are generated normally.
3. **Given** generation for one BC fails partway (e.g. the open-pencil service is down so a wireframe cannot be rendered to SVG), **When** whole-model generation runs, **Then** that BC's textual artifacts are still written (SVG omitted, scene-graph JSON still attached, warning recorded), the other BCs complete, and the run does not abort.

### User Story 5 — Declare the frontend framework and generate a system-wide frontend spec (Priority: P5)

When the user produces a downstream coding package from the PRD-generation UI, they want to **declare the frontend framework first** (Vue, React, Svelte, or another framework the catalog supports) and have the system project the full event-storming model into a frontend-shaped spec set under `specs/frontend/` — a sibling of `specs/bounded-contexts/`, not a child of any BC folder. The frontend spec captures (a) the **system-wide UI flow** in the order users actually traverse it (sequenced from cross-BC User-Story → UI bindings and Policy/Event chains, not from the per-BC random-access view that `requirements.md` provides), (b) the **menu / navigation structure** (top-level routes, nested sections, the BC each route belongs to), (c) the **chosen frontend framework declaration** with the resulting component-organization conventions, and (d) one entry per bound wireframe pointing back to the canonical scene-graph / SVG / element-tree that already lives under the owning BC's `requirements.assets/`. The per-BC `requirements.md` wireframe blocks are **not removed** — they remain the authoritative per-aggregate context — but the frontend folder is the place where a frontend engineer (or a frontend-shaped coding agent) sees "build this app, in this framework, with this navigation, in this order."

**Why this priority**: The current artifact set is BC-shaped end-to-end; a frontend engineer reading it has to manually reassemble a global navigation. Declaring the framework in the PRD UI and emitting a frontend-shaped spec set is the bridge from "DDD model" to "frontend that mirrors the user's journey through that model." It cannot ship before the per-BC artifacts are stable (P1–P4) — it consumes them — so it is sequenced after them.

**Independent Test**: From the PRD-generation UI, select frontend framework = Vue (or React, or Svelte), trigger generation against a graph with ≥2 Bounded Contexts that each have ≥1 User Story with a bound UI, and verify (a) `specs/frontend/framework.md` exists and records the declared framework + the conventions it implies (component file naming, state-management default, routing library default); (b) `specs/frontend/menu-structure.md` exists with a tree of top-level routes grouped by BC; (c) `specs/frontend/ui-flow.md` exists with an ordered narrative of UI screens that follows the cross-BC event-modeling flow, each screen entry linking back to the BC's `requirements.assets/<story>-<ui>.scene.json` and `.svg`; (d) no `.md` is written under `specs/bounded-contexts/<bc>/` whose only purpose is "frontend coordination" — those concerns live under `specs/frontend/` only; (e) regenerating with the same graph + the same framework produces byte-identical files (excluding the `Generated:` line).

**Acceptance Scenarios**:

1. **Given** the user opens the PRD-generation UI and has not yet selected a frontend framework, **When** they attempt to trigger generation with `include_frontend=true`, **Then** the UI blocks the action with a validation message naming "Select a frontend framework first (Vue / React / Svelte / …)" — the framework declaration is a precondition for frontend-spec generation, not an afterthought.
2. **Given** the user selects framework = `vue` and triggers generation, **When** the run completes, **Then** the resulting package contains `specs/frontend/framework.md` whose first non-heading line records `Framework: vue` and whose body lists the conventions the framework implies (SFC layout, Pinia or equivalent state default, Vue Router defaults); the same flow with `react` or `svelte` yields a parallel file with framework-appropriate conventions.
3. **Given** the graph models two Bounded Contexts with a cross-BC Policy → Event chain whose downstream Event triggers a User Story bound to a UI, **When** the user generates the frontend spec, **Then** `specs/frontend/ui-flow.md` lists the upstream User Story's UI *before* the downstream one, mirroring the event-modeling causal order rather than per-BC alphabetic order.
4. **Given** a User Story is bound to a UI node with a stored scene graph, **When** the frontend spec is generated, **Then** the corresponding entry in `specs/frontend/ui-flow.md` references the existing `specs/bounded-contexts/<bc>/requirements.assets/<story>-<ui>.scene.json` and `.svg` by relative path — the wireframe assets are **not duplicated** under `specs/frontend/`.
5. **Given** the graph carries no cross-BC Policy/Event flows (only one BC, or BCs that don't talk to each other), **When** the frontend spec is generated, **Then** `specs/frontend/ui-flow.md` falls back to a per-BC insertion-order sequencing and records a warning explaining the fallback; generation does not abort.

### User Story 6 — Split PRD.md (project composition) from CLAUDE.md (engineering constitution) (Priority: P6)

The currently generated `PRD.md` mixes two concerns: an overview of *what the project is* (bounded-context inventory, technology stack table, deployment view, file index) and a *constitution* (prescriptions like "you MUST read X before coding", DDD naming rules, EARS conventions, Aliases-to-AVOID enforcement, GWT-to-test requirements). The user wants these separated so that `PRD.md` is the project-composition document — readable by a product or architecture stakeholder — and `CLAUDE.md` is the AI/engineer-facing constitution — the rules every code-writing pass must honour. Whichever AI assistant is selected (Cursor or Claude), the rules content surfaces in the corresponding assistant-native file (`.cursorrules` for Cursor, `CLAUDE.md` for Claude); `PRD.md` is purely compositional.

**Why this priority**: Today both readers — product/architecture and AI/engineer — page through the same fat `PRD.md` and ignore the half that isn't for them. The split is a low-risk content reorganization (no new outputs, just relocating existing prose) that immediately improves usability for both audiences and removes drift between PRD.md and CLAUDE.md (which currently restate parts of each other).

**Independent Test**: Generate the PRD package with `ai_assistant=claude` and `spec_format=ddd`, then check that `PRD.md` contains the technology-stack table, the BC inventory table, the project-file index, and the deployment view — and **does not** contain any prescriptive "you MUST" / "before starting implementation" / "DDD principles" / "EARS notation rules" block; that prescriptive content appears in `CLAUDE.md` instead. The same check with `ai_assistant=cursor` puts the prescriptive content in `.cursorrules`, and `PRD.md` remains the same compositional file.

**Acceptance Scenarios**:

1. **Given** the generator runs with `ai_assistant=claude`, **When** the resulting package is opened, **Then** `PRD.md` contains exactly: project name, technology-stack table, Bounded-Contexts inventory table, project-structure / file index, deployment view (Docker/K8s/profile selections), and pointers to `CLAUDE.md` and per-BC artifact folders — and contains **zero** "🚨 CRITICAL" / "Before Starting Implementation" / "DDD Principles" / "EARS notation" prescriptive sections.
2. **Given** the same run, **When** `CLAUDE.md` is opened, **Then** it contains the project context + the prescriptive constitution: read-order for spec folders, ubiquitous-language enforcement, EARS-translation rules, GWT-test obligation, "do not invent domain concepts", framework conventions pointer, and references to the existing skills under `.claude/skills/`.
3. **Given** the generator runs with `ai_assistant=cursor`, **When** the resulting package is opened, **Then** `.cursorrules` carries the prescriptive constitution, `PRD.md` is unchanged in its compositional shape, and no `CLAUDE.md` is produced.
4. **Given** a section appears in both PRD.md and CLAUDE.md today (e.g. the "Reference Files" list and the per-BC inventory), **When** generation runs after this change, **Then** that content appears in exactly one file (CLAUDE.md owns the read-order prescriptions; PRD.md owns the inventory table) — no semantic duplication.

### User Story 7 — Drop per-BC agents; emit role-based agents + a frontend slash command (Priority: P7)

Today the generator emits one `.claude/agents/<bc_name>_agent.md` per Bounded Context. The user wants this removed: per-BC agents are unnecessary scaffolding — commands and skills are sufficient to convey "implement this BC". Whatever useful content currently sits inside per-BC agent files (skills-reference lists, scope-and-boundary statements, key-component summaries) is migrated into the existing skills (`.claude/skills/ddd-spec-implementation.md` and friends) and the existing slash commands (`.claude/commands/implement-ddd-bc.md`, `.claude/commands/implement-ddd-wireframe.md`). Agents are emitted **only** as role-based personas: a Frontend Engineer agent and a DDD Specialist agent, both authored once for the project (not per-BC), both whose job is to *reference* the existing skills rather than restate them. A new `.claude/commands/generate-frontend.md` slash command consumes `specs/frontend/` + each BC's `requirements.assets/` to produce the frontend app in the declared framework, and may invoke the Frontend Engineer agent.

**Why this priority**: Per-BC agent files create N redundant copies of the same skill-reference list; they also imply a sandboxing model ("agent X only touches BC X's folder") that doesn't match how cross-BC features are actually built. Collapsing to role-based agents + skills + commands is a cleanup of an already-shipped construct. It is sequenced last because it depends on the frontend-spec story (P5) — the new `/generate-frontend` command consumes that spec — and on the PRD-vs-CLAUDE split (P6) — the role-based agents reference CLAUDE.md, not PRD.md.

**Independent Test**: Generate the package with `ai_assistant=claude`, `spec_format=ddd`, `include_frontend=true`, framework = (Vue|React|Svelte). Verify (a) zero files exist at `.claude/agents/<bc_name>_agent.md`; (b) exactly two role-based agents exist at `.claude/agents/frontend-engineer.md` and `.claude/agents/ddd-specialist.md`, each of which references the skills under `.claude/skills/` by relative path rather than restating their content; (c) a new slash command at `.claude/commands/generate-frontend.md` exists whose body walks `specs/frontend/` and each BC's `requirements.assets/` and invokes the Frontend Engineer agent; (d) the existing `implement-ddd-bc.md` and `implement-ddd-wireframe.md` slash commands continue to exist and may now reference the DDD Specialist agent.

**Acceptance Scenarios**:

1. **Given** the user generates the PRD package, **When** the resulting zip is extracted, **Then** there is **no** file matching `.claude/agents/<bc_name>_agent.md` for any Bounded Context — the per-BC agent file is no longer emitted.
2. **Given** the same generation, **When** `.claude/agents/` is listed, **Then** exactly two role-based agent files exist: `frontend-engineer.md` and `ddd-specialist.md`; each one's body references `.claude/skills/ddd-principles.md`, `.claude/skills/eventstorming-implementation.md`, `.claude/skills/gwt-test-generation.md`, the chosen tech-stack skill, and (for the frontend agent) the chosen frontend skill, by relative path — not by restated content.
3. **Given** the same generation, **When** `.claude/commands/generate-frontend.md` is opened, **Then** its body reads `specs/frontend/framework.md`, `specs/frontend/menu-structure.md`, `specs/frontend/ui-flow.md`, walks each linked `requirements.assets/<story>-<ui>.scene.json` + `.svg` + element-tree block, and instructs Claude Code to produce one component per wireframe in the declared framework.
4. **Given** any *useful* prose that used to live in the per-BC agent files (skills-reference list, scope-boundary statement, key-component recap), **When** generation runs after this change, **Then** that prose appears verbatim or paraphrased inside the appropriate skill or command file — not lost — and the corresponding role-based agent merely points at it.

### Edge Cases

- A Bounded Context has Aggregates but none of their Commands carry GWT criteria: the Aggregate Specs are still produced; the "Enforced Invariants" section lists each invariant the graph *does* hold (the Aggregate's own `invariants` array, if any) and records every GWT-less Command in an "Open Decisions" note; a warning is reported.
- An Aggregate has no attributes recorded: the Properties table is rendered empty with a "no properties modeled yet" note; generation continues.
- The graph does not model external-system integrations at all: no `acl-*.md` files are produced — this is correct, not a gap (ACL is per external integration; zero integrations → zero files). A note in `bc-<slug>.md` says no external integrations are modeled.
- The graph does not record a Bounded Context's strategic classification (Core / Supporting / Generic): the Bounded Context Canvas marks it "(not classified — confirm)" and reports a warning.
- A UI node has a `template` but no `sceneGraph` (or vice versa): the wireframe section uses whatever is present; if neither, the User Story still appears in `requirements.md` with a "no wireframe modeled" note.
- The open-pencil wireframe service (`WIREFRAME_SERVICE_URL`) is unavailable: SVG rendering is skipped; the textual element tree and the attached scene-graph JSON still go into `requirements.md`; a warning is reported. Generation never aborts on this.
- Two regenerations target the same BC concurrently: the second detects the in-progress run and returns a busy response rather than corrupting the folder.
- An Aggregate name or Bounded Context name slugs to an empty or colliding string: the slug falls back to a short hash suffix so folder/file names stay unique (same rule for all artifacts).
- A Command participates in more than one User Story: its EARS criteria appear under each participating story in `requirements.md`, deduplicated by criterion id within a story.
- The user triggers PRD generation with `include_frontend=true` but has not selected a frontend framework: the UI refuses the action (FR-020) and points at the framework selector; no `specs/frontend/` is written.
- The user selects a frontend framework that the catalog does not (yet) carry conventions for: `specs/frontend/framework.md` is still emitted with the declared framework name, the conventions section is rendered with a "(no curated conventions for this framework — confirm)" marker, and a warning is reported; generation does not abort.
- The graph has only one Bounded Context, or no cross-BC Policy/Event flows: `specs/frontend/ui-flow.md` falls back to per-BC insertion order and records a warning that the sequencing is not narrative-driven (FR-021 fallback).
- A bound UI node exists in the BC's `requirements.md` but is not reachable from any cross-BC flow: it is still listed in `specs/frontend/ui-flow.md` at the tail with the label "(unreferenced flow — review)" so a frontend engineer doesn't silently lose a screen.
- A previous generation produced `.claude/agents/<bc_name>_agent.md` per-BC files; on regeneration after this feature lands, those files are **not re-emitted**. The user is responsible for deleting the stale per-BC agent files from their working copy — the generator never deletes consumer-side files (FR-013 stance), but it reports them under `skipped` with reason `deprecated_per_bc_agent`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST let a user trigger DDD-artifact generation for a single Bounded Context, identified by its Bounded Context node identifier.
- **FR-002**: The system MUST let a user trigger regeneration of the system-wide Context Map (`specs/context-map.md`) from all Bounded Contexts and the modeled flows between them.
- **FR-003**: The system MUST let a user trigger refresh of a single Aggregate Design Spec (`aggregates/aggregate-<slug>.md`), identified by Aggregate node identifier, without rewriting other artifacts.
- **FR-004**: The system MUST let a user trigger whole-model generation: the Context Map plus every Bounded Context's full artifact folder, with streamed progress.
- **FR-005**: For a generated Bounded Context, the system MUST produce, under `specs/bounded-contexts/<bc-slug>/`: `domain-terms.md` (Ubiquitous Language), `bc-<bc-slug>.md` (Bounded Context Canvas), one `aggregates/aggregate-<slug>.md` per Aggregate (Aggregate Design Spec), `requirements.md` (User Stories + EARS Acceptance Criteria + wireframes), and one `acl-<external-slug>.md` per modeled external-system integration.
- **FR-006**: `domain-terms.md` MUST contain one `## Term:` entry per Aggregate, Command, Event, ReadModel, and key Property within the BC, each with **Definition**, **Business Context**, **Related Terms**, and **Aliases to AVOID** sub-fields. Where the graph does not supply "Aliases to AVOID", the entry MUST either omit that line or carry an LLM-suggested list explicitly marked "(suggested — confirm)"; this is a configuration choice exposed on the request.
- **FR-007**: `bc-<bc-slug>.md` MUST contain the Bounded Context Canvas sections: Purpose, Strategic Classification (Domain / Business model / Evolution), Inbound Communication (table: From context | Channel | Message type | Pattern), Outbound Communication (same shape), Ubiquitous Language summary (pointer to `domain-terms.md` plus the key terms), Business Decisions, and Assumptions. Sections whose data the graph lacks MUST be rendered with an explicit "(not modeled — confirm)" marker, never silently omitted.
- **FR-008**: Each `aggregate-<slug>.md` MUST contain: Description, Aggregate Root, Member Entities and Value Objects, Properties (table: Field | Type | Mutability), Enforced Invariants (numbered, EARS notation — `WHEN…`, `IF…`, `THEN system SHALL…`, or the unconditional `THE <Aggregate> SHALL…`), Corrective Policies (eventual consistency), Commands (table: Command | Preconditions | Postconditions | Events emitted), Domain Events Emitted, and a Repository Interface stub derived from the Aggregate's identity type and Commands.
- **FR-009**: `requirements.md` MUST list the BC's User Stories grouped by Aggregate, in priority order where the graph carries priority and otherwise insertion order, each with its narrative, its Acceptance Criteria rendered in EARS notation, and — when the story is bound to a UI node — a wireframe section (textual element tree of the stored scene graph; an embedded SVG when the open-pencil service renders one; the raw scene-graph JSON attached as a sibling file).
- **FR-010**: `specs/context-map.md` MUST contain a Mermaid `graph LR` of all Bounded Contexts and their relationships, plus a Relationships section with one `### A → B` block per cross-BC edge carrying **Pattern**, **Direction**, **Translation** (or **Reason**), and a **Spec file** pointer. Where the graph does not record a relationship pattern, the block MUST use a heuristic default, mark it "(inferred — confirm)", and the operation MUST report a warning.
- **FR-011**: The system MUST translate Given/When/Then criteria from the graph into EARS notation deterministically — `Given X, When Y, Then Z` → `WHEN Y IF X THEN system SHALL Z`; a criterion with no Given → `WHEN Y THEN system SHALL Z`; an Aggregate-level unconditional invariant → `THE <Aggregate> SHALL <constraint>`. LLM smoothing of the resulting English/Korean MUST be optional and OFF-able for deterministic regeneration.
- **FR-012**: The system MUST render each bound UI wireframe from the scene graph stored in the graph (`UI.sceneGraph`), with NO call to any Figma API and NO Figma credential. Rendering MUST always produce the textual element tree and attach the raw scene-graph JSON; it MUST additionally produce an SVG image when the open-pencil wireframe service is available, and degrade gracefully (SVG omitted, warning reported) when it is not.
- **FR-013**: The system MUST refuse to overwrite an existing artifact (file or BC folder) unless the caller explicitly requests overwrite. When overwrite is requested, generated `.md` and attached `.scene.json`/`.svg` files are replaced; any attached asset no longer referenced by the new content is reported as stale (but not deleted).
- **FR-014**: The system MUST tolerate partial graph data: missing GWT, missing attributes, missing strategic classification, missing relationship patterns, missing external-system modeling, or an unavailable wireframe service must each degrade gracefully with an explicit in-document marker and a reported warning — never an aborted generation.
- **FR-015**: The system MUST produce identical artifact content for identical graph input across runs (excluding any `Generated:` timestamp line), so re-running generation against an unchanged graph is semantically a no-op.
- **FR-016**: The system MUST NOT modify the Neo4j graph as a side effect of generation. Generation is a pure projection from graph to filesystem.
- **FR-017**: The system MUST emit, per generation operation, structured logs recording which graph nodes were consumed and which warnings were raised, attached to the operation's correlation identifier; and MUST return, per operation, a structured report of created files, skipped items (with reasons), and warnings.
- **FR-018**: The system MUST NOT generate or modify `specs/constitution.md`. The project's existing constitution at `.specify/memory/constitution.md` is referenced by relative path from the generated artifacts; it is not duplicated or regenerated by this feature.
- **FR-019**: The system MUST refuse to generate a Bounded Context folder when the BC has neither Aggregates nor User Stories, returning a validation error pointing the user back to the event-storming flow.
- **FR-020**: The PRD-generation UI MUST require an explicit frontend-framework declaration (Vue / React / Svelte / other supported value) **before** any frontend artifact is produced. When `include_frontend=true` and no framework is selected, the action MUST be refused with a validation message naming the missing selection. The chosen framework name MUST be recorded verbatim in `specs/frontend/framework.md` and MUST be the basis on which all subsequent frontend conventions are projected.
- **FR-021**: When `spec_format=ddd` AND `include_frontend=true`, the system MUST produce a sibling folder `specs/frontend/` (not nested inside any Bounded Context folder) containing at minimum:
  - `framework.md` — the declared framework + its implied conventions (component file shape, default state management, default routing library, default styling approach).
  - `menu-structure.md` — a hierarchical menu / navigation tree, top-level entries grouped by Bounded Context, with each leaf naming a route and pointing at the owning BC and User Story.
  - `ui-flow.md` — a narrative ordering of UI screens that mirrors the **event-modeling causal order** of the cross-BC flow (upstream Commands → Events → Policies → downstream Commands → bound UI), with each screen entry referencing the canonical `specs/bounded-contexts/<bc>/requirements.assets/<story>-<ui>.scene.json` and `.svg` by relative path. When no cross-BC flow data is available, fall back to per-BC insertion order and record a warning.
- **FR-022**: The generated `PRD.md` MUST contain **only project-composition** content — project name, technology-stack table, Bounded-Contexts inventory, project-file index, deployment view, and pointers to companion files. The generated `PRD.md` MUST NOT contain prescriptive engineering content (read-order injunctions, DDD principles, EARS-translation rules, GWT-test obligations, "you MUST read X first" blocks). Prescriptive content MUST be emitted into `CLAUDE.md` (when `ai_assistant=claude`) or `.cursorrules` (when `ai_assistant=cursor`), and MUST NOT be duplicated across files.
- **FR-023**: The PRD-generation pipeline MUST NOT emit per-Bounded-Context agent files (`.claude/agents/<bc_name>_agent.md`). It MAY emit role-based agent files at fixed names (`.claude/agents/frontend-engineer.md`, `.claude/agents/ddd-specialist.md`), exactly one of each per project; each role-based agent's body MUST reference the existing skills under `.claude/skills/` by relative path and MUST NOT restate skill content. Useful prose from the previous per-BC agent format (skills-reference lists, scope/boundary statements) MUST be migrated into the corresponding skill or slash-command file so no guidance is lost.
- **FR-024**: When `spec_format=ddd` AND `include_frontend=true`, the system MUST emit a slash command at `.claude/commands/generate-frontend.md` whose body walks `specs/frontend/framework.md`, `specs/frontend/menu-structure.md`, and `specs/frontend/ui-flow.md`; follows each `ui-flow` entry's link to `specs/bounded-contexts/<bc>/requirements.assets/<story>-<ui>.scene.json` and `.svg` and the corresponding element-tree block in `requirements.md`; and instructs the coding agent to produce one component per wireframe in the declared framework. The command MAY invoke the Frontend Engineer role-based agent.

### Key Entities

- **Bounded Context**: The boundary within which the model and Ubiquitous Language are consistent. Carries name, description, and (if modeled) strategic classification. The unit for the BC Canvas and `domain-terms.md`.
- **Aggregate**: A consistency boundary inside a BC, with attributes (with mutability hints), invariants, Commands, Events, Policies, and ReadModels. The unit for an Aggregate Design Spec.
- **Command / Event / Policy / ReadModel**: Event-storming objects attached to an Aggregate; rendered into the Aggregate Spec's tables and into `domain-terms.md` glossary entries.
- **GWT Criterion**: A Given/When/Then acceptance criterion attached to a Command; the source for EARS invariants and EARS acceptance criteria.
- **User Story**: A user-facing requirement attached to an Aggregate within a BC; may be bound to a UI node. Rendered into `requirements.md`.
- **UI Wireframe**: A `UI` node with a stored open-pencil scene graph (`sceneGraph`), optionally a `template`, and references back to Figma (`figmaFileKey`/`figmaNodeId`, not used by this feature). Rendered into `requirements.md` as text + scene-graph JSON (+ SVG when possible).
- **External-System Integration**: A modeled link from a BC to an external system, if present in the graph. The unit for an ACL Spec.
- **DDD Artifact Folder**: The on-disk output: `specs/bounded-contexts/<bc-slug>/` containing `domain-terms.md`, `bc-<slug>.md`, `aggregates/aggregate-<slug>.md`, optional `acl-<external-slug>.md`, `requirements.md`, and `requirements.assets/`. Plus the system-level `specs/context-map.md`.
- **Frontend Spec Folder**: The on-disk output for the frontend perspective: `specs/frontend/` containing at minimum `framework.md` (declared framework + conventions), `menu-structure.md` (top-level navigation tree grouped by BC), and `ui-flow.md` (cross-BC narrative ordering of UIs with relative links back to per-BC `requirements.assets/`). A sibling of `specs/bounded-contexts/`, never nested under it.
- **Frontend Framework Declaration**: A required input the user makes in the PRD-generation UI before frontend-spec generation: one of Vue, React, Svelte, or another value the catalog supports. The chosen name is the basis for every framework-specific convention (component shape, state, routing, styling) the frontend spec and the `/generate-frontend` command emit.
- **Role-Based Agent**: A persona-shaped agent file at a fixed name (currently `frontend-engineer`, `ddd-specialist`) that references the existing skills rather than restating them. Replaces the previous per-BC agent files.
- **Slash Commands**: `.claude/commands/*.md` files the PRD-generation pipeline emits alongside the artifact set. Current set: `implement-ddd-bc.md`, `implement-ddd-wireframe.md`, and (newly added by P7) `generate-frontend.md`. The /-prefixed commands invoke role-based agents and reference the skills under `.claude/skills/`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user can go from "Bounded Context is fully event-stormed in the graph" to "DDD artifact folder exists with `domain-terms.md`, `bc-<slug>.md`, one Aggregate Spec per Aggregate, and `requirements.md`" in under one minute for a typical BC (≤ 10 Aggregates, ≤ 30 User Stories), with no Figma access involved at any point.
- **SC-002**: For a fully populated Bounded Context, every Aggregate, Command, Event, ReadModel, and User Story present in the graph appears in the corresponding generated artifact — verified by counting graph nodes against artifact sections, with zero missing.
- **SC-003**: Every Given/When/Then criterion on a Command appears as an EARS line in both the owning Aggregate's "Enforced Invariants" and the relevant User Story's "Acceptance Criteria" in `requirements.md` — verified by counting GWT nodes against EARS lines, with zero missing.
- **SC-004**: Every User Story bound to a UI node with a stored scene graph yields, in `requirements.md`, a non-empty textual element tree and an attached `.scene.json` file — verified by counting bound UI nodes against rendered wireframe blocks, with zero missing on the success path; an SVG is additionally present whenever the wireframe service was reachable.
- **SC-005**: Re-running generation against an unchanged graph (with overwrite) produces byte-identical artifact files (excluding any `Generated:` timestamp line), so reviewers detect "the graph changed" purely by diffing the output.
- **SC-006**: When the graph is partially populated (e.g. an Aggregate's Commands lack GWT, or the BC has no strategic classification), generation still produces a usable artifact set where every gap is explicitly marked in-document and named in the operation's warning list — verified by feeding a known half-populated BC and confirming each gap appears as a marker and a warning.
- **SC-007**: When the open-pencil wireframe service is unavailable, every BC's textual artifacts are still produced and every bound wireframe still gets its element tree and `.scene.json`; only the SVG images are missing, and each is reported in the warning list — verified by generating with the service stopped.
- **SC-008**: The generated artifacts conform to the "DDD for SDD" templates — `domain-terms.md` has the four sub-fields per term including "Aliases to AVOID" (present, suggested-marked, or explicitly omitted per config); each Aggregate Spec has all nine sections including "Enforced Invariants" in EARS and a "Corrective Policies" section; `context-map.md` has a Mermaid diagram and a Relationships section — verified by a structural lint over the output.
- **SC-009**: A user can declare a frontend framework (Vue / React / Svelte) in the PRD-generation UI and, in the same run that produces the DDD artifact folders, also receive a `specs/frontend/` folder with `framework.md` (declared framework + conventions), `menu-structure.md` (BC-grouped navigation tree), and `ui-flow.md` (cross-BC narrative order of UIs) — verified by listing the package contents.
- **SC-010**: For a graph with N Bounded Contexts connected by at least one cross-BC Policy/Event chain, every UI in `specs/frontend/ui-flow.md` is sequenced according to that causal chain (upstream UI before downstream UI), and every entry links to a real, present `requirements.assets/<story>-<ui>.scene.json` under the owning BC — verified by following each link from the file and confirming the target exists.
- **SC-011**: `PRD.md` and `CLAUDE.md` (or `.cursorrules`, depending on assistant selection) carry **disjoint** content — the technology-stack table and BC inventory live only in `PRD.md`; "you MUST read X first", DDD principles, EARS-translation rules, and GWT-test obligations live only in `CLAUDE.md` / `.cursorrules` — verified by a textual diff that flags any prescriptive imperative ("MUST", "SHALL", "Before starting") appearing in `PRD.md` and any inventory/stack table content appearing in the constitution file.
- **SC-012**: After this feature lands, the generated package contains **zero** files at `.claude/agents/<bc_name>_agent.md` and **exactly one each** of `.claude/agents/frontend-engineer.md` and `.claude/agents/ddd-specialist.md` whenever `ai_assistant=claude`. Each role-based agent's body contains at least one relative reference to a file under `.claude/skills/` and no restated skill content — verified by parsing the agent files and checking for skill-file references vs. restated body length.
- **SC-013**: `.claude/commands/generate-frontend.md` exists whenever `ai_assistant=claude AND spec_format=ddd AND include_frontend=true`, and a smoke run of "follow the command end-to-end" from an empty project against a known graph produces a runnable frontend project in the declared framework — verified by the manual smoke checklist (added to `quickstart.md` in P5–P7 follow-up tasks).

## Assumptions

- The Neo4j graph already contains Bounded Contexts, Aggregates (with attributes and, where modeled, invariants), Commands with GWT acceptance criteria, Events, Policies, ReadModels, User Stories, and `UI` nodes carrying open-pencil scene graphs — populated by the existing event-storming ingestion workflow and the Figma plugin sync (`UI.sceneGraph` is already persisted; see `docs/cypher/schema/03_node_types.cypher`).
- "DDD for SDD" (Jarosław Wasowski, May 2026 — `docs/DDD for SDD - …Medium.pdf`) defines the artifact formats this feature emits: `domain-terms.md` (Ubiquitous Language), `bc-{name}.md` (Bounded Context Canvas), `context-map.md` (Context Map), `aggregate-{name}.md` (Aggregate Design Spec), `acl-{system}.md` (Anti-Corruption Layer Spec), and `requirements.md` (User Stories + EARS). The output folder layout follows that article: `specs/bounded-contexts/<bc>/...` plus a system-level `specs/context-map.md`. `constitution.md` is NOT generated — the project's existing `.specify/memory/constitution.md` plays that role.
- This feature does NOT integrate with GitHub Spec Kit / SpecKit slash commands and does NOT render the SpecKit `spec.md`/`plan.md` templates. (The SpecKit artifacts that already exist under `specs/NNN-*/` — including this very feature's own `spec.md`/`plan.md` — are untouched and continue to be authored via the SpecKit workflow.)
- UI wireframes are rendered from the scene graph already stored in Neo4j; no Figma API token (`FIGMA_API_TOKEN` or otherwise) is introduced or required. The optional SVG rendering uses the existing open-pencil wireframe service at `WIREFRAME_SERVICE_URL`.
- The output language follows what the graph stores (technical identifiers in English, narrative possibly Korean); the generator does not translate. Section headings follow the article's English templates.
- Relationship patterns in the Context Map (Customer-Supplier, Conformist, ACL, Open Host Service, Published Language, Shared Kernel, Separate Ways) are largely not modeled explicitly in the graph today; the generator infers a best-guess and marks it for human confirmation, consistent with the article's stance that Context Map relationship patterns are reviewed and corrected by a human.
- ACL Specs are produced only for external-system integrations the graph actually models; with none modeled, zero ACL files is the correct output, not a gap.
- Generation is triggered manually by a user via API call (and later, via UI actions in the event-storming canvas / Aggregate inspector); it is NOT auto-triggered by graph mutations.
- Existing `specs/NNN-*/` SpecKit folders are read-only with respect to this feature; it writes only under `specs/bounded-contexts/`, `specs/frontend/`, and the single file `specs/context-map.md`, plus the consumer-side `PRD.md` / `CLAUDE.md` / `.cursorrules` / `.claude/{skills,agents,commands}/*` / `Frontend-PRD.md` that the PRD-generation flow zips into the downloaded package.
- The "DDD-for-SDD packaging" surface (Claude skills `.claude/skills/ddd-spec-implementation.md` + slash commands `.claude/commands/implement-ddd-bc.md` and `implement-ddd-wireframe.md`) already ships in the codebase under `api/features/prd_generation/`; this feature owns and continues to evolve that surface, and adds `.claude/commands/generate-frontend.md` + role-based agents `.claude/agents/{frontend-engineer,ddd-specialist}.md`. The previously emitted per-BC `.claude/agents/<bc_name>_agent.md` files are deprecated and no longer produced.
- The PRD UI is the **only** entry point for the frontend-framework declaration (FR-020). The frontend perspective is no longer "deferred to a follow-up PR" — it is in scope here as user stories P5–P7.
- The frontend artifact folder is `specs/frontend/` — a **sibling** of `specs/bounded-contexts/`. It is never written inside any BC folder, because the frontend's job is to weave UIs across BCs into a single user-journey, which is structurally orthogonal to the per-BC slicing.
- The set of supported frontend frameworks is whatever the existing PRD-generation framework catalog enumerates; the spec uses Vue / React / Svelte as the canonical examples but does not constrain to a fixed list — adding a new framework to the catalog is a downstream task, not a spec change.
