# Feature Specification: 코드에서 요구사항 역추출 (Reverse Intent)

**Feature Branch**: `047-reverse-intent`

**Created**: 2026-07-08 · **Revised**: 2026-07-08 (부분 선택으로 정정 — 이전 "DB 전체 자동" 결정 supersede)

**Status**: Draft

**Input**: 레거시 코드 분석 그래프에서, 문서 업로드 없이 코드만으로 요구사항(Strategic Diff)을 역추출하는 새 Proposal 진입 모드. 사용자는 분석된 그래프를 고른 뒤 **어떤 데이터 그룹을 요구사항으로 뽑을지 직접 선택**한다(기본값은 전체, 일부만도 가능). 검증된 별도 PoC(reverse-intent-poc, 2026-07-01)를 제품에 통합한다.

## 개요 (Why)

지금까지 architect가 "이미 있는 시스템 → 요구사항"을 만들려면 **문서(BPM Task)가 반드시 필요**했다. 그래서 "문서는 없고 레거시 코드만 있는" 상황을 다룰 수 없었다.

이 기능은 **문서 없이, 분석된 코드 그래프만으로** 요구사항(Epic/Feature/UserStory/Process)을 역추출한다. 핵심 아이디어는 "같은 데이터(저장소)를 다루는 처리 단위끼리 묶으면 그것이 하나의 업무 단위(Aggregate)가 된다"는 것이며, 별도 PoC로 열화 없이 동작함이 실측 검증되었다.

★**전체를 통째로 돌리지 않는다.** 그래프를 데이터 그룹으로 펼쳐 **사람이 읽는 카드**로 보여주고, 사용자가 **"어떤 그룹을 요구사항으로 뽑을지" 골라서** 실행한다(기본값 = 전체 선택, 원하면 일부만). 이렇게 해야 큰 레거시에서 필요한 부분만 점진적으로 요구사항화할 수 있다.

이 기능은 기존 Proposal 흐름을 대체하지 않는다. 새 "종류(모드)"를 하나 더 추가할 뿐이며, 만들어진 요구사항은 기존과 동일한 형태라 이후 설계·구현 단계를 그대로 탄다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 문서 없이, 고른 데이터에서 요구사항 도출 (Priority: P1)

사용자가 새 Proposal을 만들 때 종류로 "코드에서 역추출"을 고르고 분석된 그래프를 선택하면, 시스템이 그 코드를 **데이터 그룹으로 펼쳐 보여준다**. 사용자는 **뽑고 싶은 그룹을 선택**(기본 전체, 일부 해제 가능)한 뒤 실행하면, 자연어 요구사항 문장을 한 줄도 쓰지 않고 **선택한 그룹만** 요구사항(Epic/Feature/UserStory) 초안으로 받는다.

**Why this priority**: 이 기능의 핵심 가치. 문서 없는 레거시를 다루되, 통째가 아니라 **필요한 부분만 골라** 요구사항화하는 것이 실무 요구다(MVP).

**Independent Test**: 분석된 그래프를 선택 → 그룹 목록이 뜨고 → 일부만 체크해 실행 → 그 그룹들에 해당하는 요구사항만 생성되는지로 검증.

**Acceptance Scenarios**:

1. **Given** 분석된 그래프를 선택했을 때, **When** 화면이 로드되면, **Then** 데이터 그룹들이 **선택 가능한 카드 목록**(기본 전체 선택)으로 표시된다.
2. **Given** 일부 그룹만 선택했을 때, **When** 실행하면, **Then** **선택한 그룹의 처리 단위만** 요구사항으로 도출되고 선택 안 한 그룹은 제외된다.
3. **Given** 전체 그룹을 선택(기본값)했을 때, **When** 실행하면, **Then** 그래프 전체가 요구사항으로 도출된다.
4. **Given** 역추출이 완료되었을 때, **When** 이후 설계(Plan) 단계로 진행하면, **Then** 추가 수작업 없이 기존 Proposal 흐름과 동일하게 이어진다.

---

### User Story 2 - 헤매지 않게, 읽고 고르는 그룹 카드 (Priority: P1)

그룹이 기술적 식별자가 아니라 **사람이 읽는 업무 이름·성격·포함 작업**으로 표시되어, 사용자가 "이게 뭔지" 헤매지 않고 무엇을 선택할지 즉시 이해한다. 선택 UI는 기존 Proposal 화면과 일관되고, 정렬·색상·접근성이 깔끔해야 한다.

**Why this priority**: 선택하려면 먼저 이해해야 한다. 카드가 안 읽히면 선택(US1) 자체가 불가능하므로 P1(선택과 한 몸).

**Independent Test**: 그룹 카드가 업무 이름(제목)·성격 배지·작업 목록으로 렌더되고, 행마다 요소 위치가 어긋나지 않으며(지그재그 0), 라이트/다크 배경에 맞는 색으로 표시되는지 스크린샷으로 검증.

**Acceptance Scenarios**:

1. **Given** 대상 그래프가 의미분석까지 완료되어 자연어 라벨을 가질 때, **When** 그룹이 표시되면, **Then** 각 카드는 업무 이름·작업 성격 배지·포함 작업 수·작업들의 자연어 설명으로 표시된다.
2. **Given** 그룹 카드 목록을 볼 때, **When** 행을 훑으면, **Then** 체크박스·제목·배지·작업수의 가로 위치가 모든 행에서 일정하게 정렬된다(요소 유무로 밀리지 않음).
3. **Given** 라이트/다크 테마 각각에서, **When** 카드를 보면, **Then** 배경·글자·배지 색이 테마에 맞고 대비가 충분하다.

---

### User Story 3 - 사이즈 무관·무손실·무중복 (Priority: P3)

수백 개 규칙을 가진 큰 시스템이어도, 선택한 그룹 전체가 빠짐없이·중복 없이 하나의 요구사항 집합으로 수렴한다.

**Why this priority**: 실제 레거시는 크다. 대용량에서 열화·누락·중복이 생기면 신뢰를 잃는다. 단, 소규모에서 먼저 가치가 증명되므로 P3.

**Independent Test**: 규칙 수가 많은 그룹을 선택해 실행하고, 그 그룹의 모든 처리 단위가 정확히 한 번 포함되고 결과에 중복이 없는지로 검증.

**Acceptance Scenarios**:

1. **Given** 처리 한도를 넘는 큰 그룹을 선택했을 때, **When** 역추출하면, **Then** 처리 단위 경계에서만 분할되고 단위 내부의 규칙은 쪼개지지 않는다.
2. **Given** 여러 그룹을 선택했을 때, **When** 최종 요구사항으로 합치면, **Then** 그룹 간 중복 업무 항목이 제거되어 하나의 일관된 집합이 된다.

---

### Edge Cases

- **선택 0개**: 아무 그룹도 선택 안 하면 실행 버튼이 비활성(또는 명확한 안내). 빈 실행 금지.
- **의미분석 미완료 그래프**: 자연어 라벨이 비면 원 이름·스키마 설명으로 폴백하고 가독성 저하를 알린다.
- **인프라/디버그/임시 저장소**: 업무 단위가 아닌 저장소는 그룹 후보에서 제외한다.
- **쓰기 없는 조회 전용 처리 단위**: 읽기 기준 그룹 또는 로직 그룹으로 배정한다(누락 금지).
- **여러 저장소에 쓰는 처리 단위**: 대표 저장소 한 곳의 그룹에만 배정된다(중복 배정 금지).
- **분석된 그래프가 하나도 없음**: 선택할 대상이 없음을 명확히 안내한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 새 Proposal의 종류로 "코드에서 역추출"을 선택할 수 있어야 한다(기존 종류 선택에 추가).
- **FR-002**: 이 모드에서 시스템은 자연어 문장 입력을 요구하지 않고, 이미 분석된 그래프 중 하나를 선택하게 해야 한다.
- **FR-003**: 시스템은 선택 가능한 "이미 분석된 그래프" 목록을 자동으로 제시해야 한다(식별자 암기 불필요).
- **FR-004**: 그래프 선택 후, 시스템은 **요구사항 도출 전에** 데이터 그룹 목록을 **선택 가능한 형태로 미리 보여줘야** 한다(LLM 호출 없이 빠르게).
- **FR-005**: 그룹 선택의 **기본값은 전체 선택**이며, 사용자는 개별 그룹을 해제해 **일부만** 실행하거나 전체를 실행할 수 있어야 한다.
- **FR-006**: 시스템은 각 처리 단위를, 그것이 데이터를 쓰는 대표 저장소를 기준으로 정확히 하나의 그룹에만 배정해야 한다(선택 단위 = 그룹).
- **FR-007**: 시스템은 인프라·디버그·임시 성격의 저장소를 그룹 후보에서 제외해야 한다.
- **FR-008**: 시스템은 각 그룹을 사람이 읽을 수 있는 형태(업무 이름·작업 성격·작업 수·포함 작업의 자연어 설명)로 제시해야 한다.
- **FR-009**: 시스템은 **선택된 그룹**의 업무 규칙·시나리오를 손실·요약 없이 요구사항 도출 입력으로 전달해야 한다.
- **FR-010**: 도출 입력이 처리 한도를 넘으면 시스템은 처리 단위 경계에서만 분할해야 하며, 한 단위 내부의 규칙을 쪼개서는 안 된다.
- **FR-011**: 시스템은 선택된 각 그룹으로부터 요구사항(Epic/Feature/UserStory/Process)을 생성해야 한다.
- **FR-012**: 시스템은 여러 그룹의 결과를 하나의 요구사항 집합으로 중복 없이 합쳐야 한다.
- **FR-013**: 생성된 요구사항은 기존 자연어 기반 Proposal 결과와 동일한 형태여야 하며, 이후 설계·구현 단계를 기존과 동일하게 진행할 수 있어야 한다.
- **FR-014**: 대상 그래프에 자연어 라벨이 없으면 시스템은 대체 라벨(원 이름/스키마 설명)로 폴백하고, 가독성 저하 가능성을 사용자에게 알려야 한다.
- **FR-015**: 시스템은 역추출 진행 상황(선택된 그룹의 순차 처리)을 사용자에게 실시간으로 보여줘야 한다.
- **FR-016**: 시스템은 분석 그래프를 읽기 전용으로만 사용해야 하며, 그래프를 변경해서는 안 된다.

### Non-Functional Requirements — UI 품질 (하드 요구)

사용자는 화면 미관·일관성에 매우 까다롭다. 아래는 "있으면 좋은"이 아니라 **충족 필수**다.

- **NFR-UI-1 (일관성)**: 선택 UI는 기존 Proposal 화면(종류 라디오·상세 트랙·결과 뷰)과 **같은 컴포넌트 스타일·색·간격 규칙**을 따른다. 같은 의미는 어디서나 같은 표현.
- **NFR-UI-2 (정렬·지그재그 0)**: 그룹 카드 목록은 **행마다 요소(체크박스·제목·배지·작업수)의 가로 위치가 일정**해야 한다. 옵션 요소 유무로 옆 요소가 밀리면 안 된다(고정 트랙/그리드, 매직 px 금지).
- **NFR-UI-3 (테마)**: 라이트/다크 **배경에 맞는 색**(테마 토큰 사용), 대비 충분. 하드코딩 색 금지.
- **NFR-UI-4 (접근성·명료성)**: 사용자가 "이게 뭐지" 헤매지 않게 — 명확한 라벨·상태(선택/해제·진행), 키보드 포커스, 불필요한 난해함 0.
- **NFR-UI-5 (검증)**: 구현 후 Playwright 실물 캡처로 정렬·일관·테마·가독성 확인.

### Key Entities *(include if feature involves data)*

- **분석된 그래프(소스)**: 레거시 코드 분석 결과(읽기 전용). 처리 단위·규칙·시나리오·저장소 관계·자연어 라벨 포함.
- **데이터 그룹(Aggregate 후보)**: 같은 대표 저장소 중심으로 묶인 처리 단위 집합. 업무 이름·성격·종류·작업 목록 + **선택 상태**를 가진다. 선택 단위.
- **요구사항 묶음(Strategic Diff)**: 선택된 그룹에서 도출된 Epic/Feature/UserStory/Process. 기존 Proposal 결과와 동일 형태.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 자연어 문장을 한 줄도 입력하지 않고, 그래프 선택 + 그룹 선택만으로 요구사항 초안을 얻는다.
- **SC-002**: 생성된 UserStory가 **선택된 그룹의** 원본 처리 단위와 1:1로 대응한다 — 환각 0, 누락 0, 중복 0.
- **SC-003**: 그룹 카드를 처음 본 사용자가 도메인 사전 설명 없이도 각 그룹이 "무슨 업무인지" 업무 이름만으로 인지하고, 무엇을 선택할지 판단한다.
- **SC-004**: 선택한 그룹의 모든 처리 단위가 정확히 한 번 포함되며(선택 범위 내 커버리지 100%), 최종 요구사항 집합의 그룹 간 중복은 0이다.
- **SC-005**: 역추출로 생성된 요구사항이 이후 설계·구현 단계로 추가 수작업 없이 그대로 이어진다.
- **SC-006 (UI)**: 그룹 카드 목록에서 행 간 요소 가로 위치 어긋남 0(지그재그 0), 라이트·다크 양쪽에서 대비·가독성 정상 — 실물 캡처로 확인.

## Assumptions

- 대상 코드는 사전에 analyzer로 **의미분석까지 완료**되어 그래프가 존재한다. 이 기능은 분석을 수행하지 않고 이미 만들어진 그래프를 소비한다.
- 그룹·작업의 자연어 라벨 품질은 의미분석 산출물에 의존한다. 라벨이 없으면 폴백하되, 정상 입력은 "의미분석 완료 그래프".
- 범위는 **요구사항(Strategic Diff) 생성까지**이며, 이후 설계·구현은 기존 Proposal 파이프라인을 재사용한다.
- 새 모드는 기존 Proposal의 종류 선택 UI에 추가되고, 결과 표시·이후 단계 화면은 기존 자산을 재사용한다.
- **선택 단위 = 데이터 그룹(Aggregate/테이블). 기본값 = 전체 선택. 사용자는 일부만 또는 전체를 실행할 수 있다.** (이전 "DB 전체 자동, 세밀 선택 비강제" 가정은 이 개정으로 대체됨.)
- 이 기능은 진행 중인 다른 작업(레거시 그래프 계약 슬림화)과 파일이 겹치지 않으며, 별도 통합 단위로 진행한다.
