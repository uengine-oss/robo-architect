# Feature Specification: 코드에서 요구사항 역추출 (Reverse Intent)

**Feature Branch**: `047-reverse-intent`

**Created**: 2026-07-08

**Status**: Draft

**Input**: 레거시 코드 분석 그래프에서, 문서 업로드 없이 코드만으로 요구사항(Strategic Diff)을 역추출하는 새 Proposal 진입 모드. 검증 완료된 별도 PoC(reverse-intent-poc, 2026-07-01)를 제품에 통합한다.

## 개요 (Why)

지금까지 architect가 "이미 있는 시스템 → 요구사항"을 만들려면 **문서(BPM Task)가 반드시 필요**했다(문서가 요구사항의 골격이고, 코드는 규칙 재료로만 붙는다). 그래서 "문서는 없고 레거시 코드만 있는" 상황을 다룰 수 없었다.

이 기능은 **문서 없이, 분석된 코드 그래프만으로** 요구사항(Epic/Feature/UserStory/Process)을 역추출한다. 핵심 아이디어는 "같은 데이터(저장소)를 다루는 처리 단위끼리 묶으면 그것이 하나의 업무 단위(Aggregate)가 된다"는 것이며, 별도 PoC로 열화 없이 동작함이 실측 검증되었다(예: 인증 이력 테이블 7개 처리 단위 → UserStory 7개, 누락·환각·중복 0).

이 기능은 기존 Proposal 흐름을 대체하지 않는다. 새로운 "종류(모드)"를 하나 더 추가할 뿐이며, 만들어진 요구사항은 기존과 동일한 형태라 이후 설계·구현 단계를 그대로 탄다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 문서 없이 코드에서 요구사항 도출 (Priority: P1)

사용자가 새 Proposal을 만들 때 종류로 "코드에서 역추출"을 고르고, 이미 분석된 그래프 하나를 선택하면, 자연어 요구사항 문장을 한 줄도 쓰지 않고 요구사항(Epic/Feature/UserStory) 초안을 받는다.

**Why this priority**: 이 기능의 핵심 가치 그 자체. 문서가 없는 레거시를 다룰 수 있게 하는 유일한 진입점이며, 이것만 있어도 독립적으로 가치를 준다(MVP).

**Independent Test**: 분석이 끝난 그래프를 하나 선택하고 실행하면, 원본 코드의 업무 규칙이 반영된 Epic/Feature/UserStory 목록이 생성되는지로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 분석이 완료된 코드 그래프가 존재하고 사용자가 "코드에서 역추출" 모드를 선택했을 때, **When** 분석된 그래프 하나를 선택해 실행하면, **Then** 자연어 입력 없이 Epic/Feature/UserStory로 구성된 요구사항 초안이 생성된다.
2. **Given** 역추출이 완료되었을 때, **When** 결과를 확인하면, **Then** 각 UserStory가 원본 코드의 처리 단위·업무 규칙과 대응된다(임의로 지어낸 항목 없음).
3. **Given** 역추출이 완료되었을 때, **When** 이후 설계(Plan) 단계로 진행하면, **Then** 추가 수작업 없이 기존 Proposal 흐름과 동일하게 이어진다.

---

### User Story 2 - 사람이 읽을 수 있는 데이터 그룹으로 확인 (Priority: P2)

역추출 과정에서 시스템이 묶은 "데이터 그룹"이 기술적 식별자(예: 테이블 코드명)가 아니라 사람이 읽는 업무 이름과 작업 목록으로 표시되어, 사용자가 무엇으로부터 요구사항이 만들어지는지 이해한다.

**Why this priority**: 결과에 대한 신뢰와 이해를 준다. 없어도 요구사항은 나오지만(P1), 있으면 사용자가 "이게 뭔지 알겠다" 하고 검토·수용할 수 있다.

**Independent Test**: 역추출 실행 시 표시되는 그룹 카드가 업무 이름(예: "자동납부 실시간 인증 이력")과 포함 작업의 자연어 설명으로 나타나는지로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 대상 그래프가 의미분석까지 완료되어 자연어 라벨을 가질 때, **When** 그룹이 표시되면, **Then** 각 그룹은 업무 이름(제목)·작업 성격 배지·포함 작업 수·작업들의 자연어 설명 목록으로 표시된다.
2. **Given** 한 그룹의 작업 이름이 그룹 제목과 겹치는 접두어를 포함할 때, **When** 목록을 보면, **Then** 중복 접두어가 정리되어 핵심 내용이 눈에 띈다.

---

### User Story 3 - 사이즈 무관·무손실·무중복 수렴 (Priority: P3)

수백 개 규칙을 가진 큰 시스템이어도 전체 처리 단위가 빠짐없이, 중복 없이 하나의 요구사항 집합으로 수렴한다.

**Why this priority**: 실제 레거시는 크다. 대용량에서 열화·누락·중복이 생기면 신뢰를 잃는다. 단, 소규모에서 먼저 가치가 증명되므로 P3.

**Independent Test**: 규칙 수가 많은 그래프 전체를 실행해, 모든 처리 단위가 어느 그룹엔가 정확히 한 번 포함되고 최종 결과에 중복이 없는지로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 처리 한도를 넘는 큰 그룹이 있을 때, **When** 역추출하면, **Then** 처리 단위 경계에서만 분할되고 단위 내부의 규칙은 쪼개지지 않는다.
2. **Given** 여러 그룹이 만들어졌을 때, **When** 최종 요구사항으로 합치면, **Then** 그룹 간 중복 업무 항목이 제거되어 하나의 일관된 집합이 된다.

---

### Edge Cases

- **의미분석 미완료 그래프**: 자연어 라벨(업무 이름·작업 설명)이 비어 있으면, 원 이름·스키마 설명으로 폴백하고 가독성이 낮을 수 있음을 사용자에게 알린다.
- **인프라/디버그/임시 저장소**: 디버그 로그·임시 테이블처럼 업무 단위가 아닌 저장소는 그룹 후보에서 제외한다.
- **쓰기 없는 조회 전용 처리 단위**: 데이터를 쓰지 않는 처리 단위는 읽기 기준 그룹 또는 로직 그룹으로 배정한다(누락 금지).
- **여러 저장소에 쓰는 처리 단위**: 한 처리 단위가 여러 저장소에 쓰더라도 대표 저장소 한 곳의 그룹에만 배정된다(중복 배정 금지).
- **분석된 그래프가 하나도 없음**: 선택할 대상이 없음을 명확히 안내한다.
- **대상 그래프 접근 불가/비어 있음**: 조용히 빈 결과를 반환하지 않고 명시적으로 알린다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 새 Proposal의 종류로 "코드에서 역추출"을 선택할 수 있어야 한다(기존 종류 선택에 추가되는 형태).
- **FR-002**: 이 모드에서 시스템은 자연어 요구사항 문장 입력을 요구하지 않고, 이미 분석된 그래프 중 하나를 선택하게 해야 한다.
- **FR-003**: 시스템은 선택 가능한 "이미 분석된 그래프" 목록을 자동으로 제시해야 한다(사용자가 식별자를 외워 입력하지 않도록).
- **FR-004**: 시스템은 선택된 그래프의 모든 처리 단위를 빠짐없이 대상으로 삼아야 한다(커버리지 100%).
- **FR-005**: 시스템은 각 처리 단위를, 그것이 데이터를 쓰는 대표 저장소를 기준으로 정확히 하나의 그룹에만 배정해야 한다.
- **FR-006**: 시스템은 인프라·디버그·임시 성격의 저장소를 그룹 후보에서 제외해야 한다.
- **FR-007**: 시스템은 각 그룹을 사람이 읽을 수 있는 형태(업무 이름·작업 성격·작업 수·포함 작업의 자연어 설명)로 제시해야 한다.
- **FR-008**: 시스템은 각 그룹의 업무 규칙과 시나리오를 손실·요약 없이 요구사항 도출 입력으로 전달해야 한다.
- **FR-009**: 도출 입력이 처리 한도를 넘으면 시스템은 처리 단위 경계에서만 분할해야 하며, 한 단위 내부의 규칙을 쪼개서는 안 된다.
- **FR-010**: 시스템은 각 그룹으로부터 요구사항(Epic/Feature/UserStory/Process)을 생성해야 한다.
- **FR-011**: 시스템은 여러 그룹의 결과를 하나의 요구사항 집합으로 중복 없이 합쳐야 한다.
- **FR-012**: 생성된 요구사항은 기존 자연어 기반 Proposal 결과와 동일한 형태여야 하며, 이후 설계·구현 단계를 기존과 동일하게 진행할 수 있어야 한다.
- **FR-013**: 대상 그래프에 자연어 라벨이 없으면 시스템은 대체 라벨(원 이름/스키마 설명)로 폴백하고, 가독성 저하 가능성을 사용자에게 알려야 한다.
- **FR-014**: 시스템은 역추출 진행 상황(여러 그룹의 순차 처리)을 사용자에게 실시간으로 보여줘야 한다.
- **FR-015**: 시스템은 분석 그래프를 읽기 전용으로만 사용해야 하며, 그래프를 변경해서는 안 된다.

### Key Entities *(include if feature involves data)*

- **분석된 그래프(소스)**: 레거시 코드 분석 결과. 처리 단위, 업무 규칙, 시나리오, 데이터 저장소(테이블)와의 읽기/쓰기 관계, 자연어 라벨을 포함한다. 이 기능의 입력이며 읽기 전용이다.
- **데이터 그룹(Aggregate 후보)**: 같은 대표 저장소를 중심으로 묶인 처리 단위 집합. 업무 이름, 성격, 종류(핵심 데이터/조회/로직), 포함 작업 목록을 가진다.
- **요구사항 묶음(Strategic Diff)**: 역추출로 생성된 Epic/Feature/UserStory/Process. 기존 Proposal 결과와 동일한 형태이며 이후 단계의 입력이 된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 자연어 요구사항 문장을 한 줄도 입력하지 않고, 분석된 그래프 선택만으로 요구사항 초안을 얻는다.
- **SC-002**: 생성된 UserStory가 원본 처리 단위와 1:1로 대응한다 — 임의로 지어낸 항목(환각) 0, 원본 처리 단위 누락 0, 중복 0. (PoC 실측 기준: 처리 단위 7개 → UserStory 7개)
- **SC-003**: 그룹 카드를 처음 본 사용자가 도메인 사전 설명 없이도 각 그룹이 "무슨 업무인지" 업무 이름만으로 인지한다.
- **SC-004**: 대상 시스템의 크기와 무관하게 모든 처리 단위가 어느 그룹엔가 정확히 한 번 포함되며(커버리지 100%), 최종 요구사항 집합의 그룹 간 중복은 0이다.
- **SC-005**: 역추출로 생성된 요구사항이 이후 설계·구현 단계로 추가 수작업 없이 그대로 이어진다(기존 파이프라인 무변경 통과).

## Assumptions

- 대상 코드는 사전에 analyzer로 **의미분석까지 완료**되어 그래프가 존재한다. 이 기능은 코드 분석 자체를 수행하지 않고, 이미 만들어진 그래프를 소비한다.
- 그룹·작업의 자연어 라벨 품질은 분석(의미분석) 산출물에 의존한다. 라벨이 없으면 폴백하되, 정상 입력은 "의미분석 완료 그래프"로 전제한다.
- 이 기능의 범위는 **요구사항(Strategic Diff) 생성까지**이며, 이후 설계(Plan)·작업분해·구현은 기존 Proposal 파이프라인을 재사용한다(변경 없음).
- 새 모드는 기존 Proposal의 종류 선택 UI에 추가되고, 결과 표시와 이후 단계 화면은 기존 자산을 재사용한다.
- 선택 단위는 "분석된 그래프 하나(스코프)"이며, 그보다 세밀한 선택은 강제하지 않는다(기본은 그래프 전체 자동 처리).
- 이 기능은 진행 중인 다른 작업(레거시 그래프 계약 슬림화)과 파일이 겹치지 않으며, 별도 통합 단위로 진행한다.
