# Feature Specification: analyzer↔architect 그래프 계약 정합 (소비자 마이그레이션 완성)

**Feature Branch**: `044-analyzer-graph-contract-realign`

**Created**: 2026-06-26

**Status**: Draft

**Input**: User description: "analyzer↔architect Neo4j 그래프 계약 정합 — 소비자(architect) 속성명/엣지/라벨을 생산자(analyzer) 실제 출력에 맞춰 진행중 마이그레이션 완성 및 조용한 NULL 유실 버그(local_rule_id↔local_id 등) 정정"

## 배경 (Context)

architect는 자기 설계 산출물을 만들기 위해, 별도 분석기(analyzer)가 레거시 소스코드를 분석해 Neo4j에 적재한 **설계 그래프**를 읽어 들인다(BPMN↔레거시 BL 룰 매핑, traceability, PRD 생성, glossary 추출, event-storming 승격 등). 즉 analyzer는 그래프의 **생산자**, architect는 **소비자**다. 두 레포는 서로 다른 스펙 체계를 가지며, 공유되는 유일한 계약면이 이 Neo4j 그래프다.

생산자(analyzer)는 이미 노드 식별자·라벨·엣지·속성 체계를 정규화 완료했다(단일 `id` 식별자, UPPER_SNAKE 단일 antlr-타입 라벨, 통합 엣지 `HAS_MEMBER`/`BELONGS_TO`, "빈값 미저장" 속성 규칙). 소비자(architect)는 이 정규화를 따라잡는 마이그레이션이 **진행 중이며 미완성**이다. 라벨/식별자 일부는 정렬됐으나, **여러 소비 지점이 생산자가 실제로 쓰지 않는 속성명을 읽어 조용히 NULL을 받는다**(에러·충돌 없이 데이터만 누락). 이 명세는 그 소비자 측 정합을 완성하는 것이 목표다.

**이 명세의 경계**: 수정 대상은 **architect(소비자)뿐**이다. analyzer(생산자)의 출력 계약은 변경 불가한 기준(ground truth)으로 간주한다.

**문제의 본질(핵심)**: 생산자 정규화로 **노드 라벨명·노드/엣지 속성명·대소문자 규칙**이 모두 바뀌었다(예: PascalCase `:Rule`/`:Table` → UPPER_SNAKE `:RULE`/`:TABLE`, `function_id`/`example_id`/`fqn` → 단일 `id`, `local_id` → `local_rule_id`, `moduleStereotype` → `stereotype`). 소비자가 옛 이름/대소문자를 그대로 쓰면 매칭이 0건이 되거나 값이 NULL이 된다. 본 작업은 소비자의 **모든 그래프 접점**을 생산자 현행 명명규칙(라벨·속성명·대소문자)에 일치시키는 것이다.

**전략 커버리지(필수)**: analyzer는 분석 **전략별**로 그래프를 만든다 — **framework**(애플리케이션 코드: 파일/클래스/메서드/함수, 코드에서 추출한 룰) 와 **dbms**(DB 객체: 스키마/테이블/컬럼/프로시저/트리거/SQL 구문, 프로시저 본문에서 추출한 룰). 소비자 정합은 **두 전략 모두**에서 성립해야 한다. 한 전략에서만 맞고 다른 전략에서 깨지면 안 된다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 룰 실행흐름(NEXT/BRANCH) 주소체계 복구 (Priority: P1)

architect가 레거시 코드에서 추출한 비즈니스 룰을 인제스천할 때, 각 룰은 "어느 함수의 몇 번째 지역 룰인가"라는 `(함수, 지역_룰_식별자)` 주소로 후속 단계(BPMN 매핑, event-storming 승격)에서 참조된다. 또한 룰 사이의 순차(NEXT)·분기(BRANCH) 흐름도 같은 지역 식별자로 연결된다. 현재 소비자는 생산자가 실제로 쓰는 흐름 식별자 속성명과 **다른 이름**을 읽어, 흐름 연결 정보가 **항상 비어 있고** 지역 식별자가 항상 비어, 룰 흐름·주소체계가 통째로 무력화된다.

**Why this priority**: 이 기능은 현재 **완전 고장(silent)** 상태다. 룰 흐름·주소가 비면 BPMN 활동↔룰 매핑·승격의 정확도가 직접 저하되며, 에러가 안 나서 발견조차 어렵다. 계약 불일치 중 영향이 가장 크고 명백하다.

**Independent Test**: 룰을 가진 레거시 함수가 적재된 그래프에 대해 인제스천을 돌려, 추출된 룰 레코드의 지역 식별자와 NEXT/BRANCH 흐름 배열이 **비어 있지 않고** 생산자가 적재한 흐름 구조와 일치하는지 확인하면 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 생산자가 한 함수에 여러 룰과 그 사이 순차/분기 흐름을 적재한 그래프, **When** architect가 룰 인제스천을 수행하면, **Then** 각 룰의 지역 식별자가 생산자 적재값과 동일하게 채워진다(NULL 아님).
2. **Given** 룰 A→B 순차(NEXT)와 A→C 분기(BRANCH)가 적재된 그래프, **When** 룰 인제스천을 수행하면, **Then** 룰 A의 next/branch 흐름 배열이 각각 B·C의 지역 식별자를 담는다(빈 배열 아님).
3. **Given** 동일 그래프, **When** 인제스천 산출 룰 식별자를 생성하면, **Then** 식별자에 `None`/빈값이 섞이지 않는다.

### User Story 2 - 생산자가 실제로 내보내는 속성만 소비 (조용한 NULL 유실 제거) (Priority: P2)

architect의 여러 소비 지점은 생산자가 (a) 이름이 다르거나 (b) 더 이상 내보내지 않거나 (c) 애초에 내보낸 적 없는 노드/엣지 속성을 읽는다. 결과적으로 해당 값이 항상 NULL/기본값이 되어, 그 값에 의존하는 화면·문서·매핑이 조용히 빈 채로 나온다. 모든 소비 지점이 **생산자의 실제 출력 속성에 정렬**되거나, 정렬 불가한 읽기는 **명시적으로 제거**되어야 한다.

**Why this priority**: P1만큼 전면적이진 않지만, 경계 예시 표시·파일명 표시·가드/분기 메타 등 여러 부수 정보가 조용히 누락된다. 누적되면 산출물 신뢰도를 떨어뜨린다.

**Independent Test**: 소비자 그래프 읽기 각 지점에 대해, 읽는 속성 키 집합이 생산자가 그 노드/엣지 종류에 실제 기록하는 속성 키 집합의 부분집합인지 대조하면 검증 가능(불일치 0건이 목표).

**Acceptance Scenarios**:

1. **Given** 생산자가 경계 여부 플래그를 노드 속성으로 내보내지 않는다는 사실, **When** 소비자가 예시(Example)의 경계 여부를 사용하면, **Then** 소비자는 NULL을 기본값으로 가리는 대신 생산자 계약에 맞는 값(또는 의도적 부재)을 일관되게 처리한다.
2. **Given** 생산자가 함수 노드에 단일 파일 경로 속성만 기록한다는 사실, **When** 소비자가 함수의 파일 정보를 읽으면, **Then** 생산자가 실제 기록하는 속성명으로 읽어 값을 얻는다(NULL 아님).
3. **Given** 생산자가 함수 식별을 단일 `id`로 통일했다는 사실, **When** 소비자가 함수를 식별/조회하면, **Then** 생산자가 기록하지 않는 레거시 식별 속성에 의존하지 않는다.

### User Story 3 - 죽은 그래프 패턴 정리 + 계약 문서를 단일 진실로 정렬 (Priority: P3)

소비자에는 생산자가 **존재시키지 않는** 노드/엣지 패턴을 향한 읽기가 남아 있다(예: 특정 행위자-역할 관계, 옛 컨테이너 식별 속성 기반 폴백 경로). 이들은 항상 무효이므로 제거하거나 유효 패턴으로 교체해야 한다. 또한 architect가 보유한 계약 문서가 **자기 코드와도, 생산자 실제 출력과도** 어긋나 있어(라벨/식별자/멤버십 엣지 명칭이 모두 불일치), 향후 재발의 근원이 된다. 계약 문서를 생산자의 실제 출력 + 소비자의 실제 동작과 일치하는 **단일 권위 문서**로 정렬한다.

**Why this priority**: 기능 데이터 흐름을 즉시 깨뜨리진 않지만(죽은 패턴은 이미 무효), 코드 정리와 문서-코드 정합으로 **재발 방지**라는 근본 가치를 제공한다.

**Independent Test**: (a) 소비자에 생산자가 만들지 않는 노드/엣지 패턴을 향한 읽기가 0건인지, (b) 계약 문서의 라벨/식별자/엣지 명세가 생산자 실제 출력 및 소비자 실제 코드와 일치하는지 문서-코드 대조로 검증 가능.

**Acceptance Scenarios**:

1. **Given** 생산자가 특정 행위자-역할 관계를 그래프에 만들지 않는다는 사실, **When** 소비자 코드를 점검하면, **Then** 해당 무효 패턴을 향한 읽기가 존재하지 않는다.
2. **Given** 계약 문서, **When** 문서의 노드 라벨·식별자·멤버십 엣지 명세를 생산자 실제 출력 및 소비자 실제 쿼리와 대조하면, **Then** 세 가지가 모두 일치한다.

### User Story 4 - dbms 깊은 구조에서 "오퍼레이션 단위" 복원 (Priority: P1)

생산자는 framework·dbms 두 전략에서 비즈니스로직(Rule/Example/Question)을 **동일한 방식**으로 붙이지만, **"오너 단위"가 다르다**: framework는 함수/메서드 1개 노드가 룰을 직접 소유하고, dbms는 프로시저가 **여러 자식 구문(SELECT/IF/LOOP/CALL…)으로 분해**되어 룰·테이블접근이 **자식 구문들에 분산**된다(자식 구문은 이름 `""`). 소비자는 "한 함수=룰 오너"를 가정하므로, dbms에서는 룰 오너가 이름 없는 구문으로 잡혀 (a) 오퍼레이션 이름이 비고, (b) 한 프로시저의 룰이 여러 조각으로 흩어지고, (c) 매칭 근거가 되는 요약이 좁은 구문 설명으로 떨어진다. 소비자는 **전략과 무관하게 "룰이 속한 가장 가까운 호출가능 루틴"을 오퍼레이션 단위로 복원**해야 한다.

**Why this priority**: framework에서는 무난히 동작하지만 **dbms 분석 결과는 사실상 소비 불가**(오퍼레이션이 안 잡혀 user story/command 생성·매칭이 붕괴). 두 전략 동시 지원이 본 작업의 핵심 목표(FR-011)이므로 P1.

**Independent Test**: dbms 전략으로 분석·적재한 그래프(프로시저+자식구문+룰)에 대해 소비를 재실행해, 각 룰이 **자식 구문이 아니라 소유 루틴(프로시저) 단위로 묶이고**, 그 루틴의 이름·요약으로 식별/매칭되는지 확인하면 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 한 프로시저가 여러 자식 구문으로 분해되고 룰이 그 자식들에 붙은 dbms 그래프, **When** 소비자가 룰을 처리하면, **Then** 룰들이 **그 프로시저(루틴) 하나의 오퍼레이션**으로 묶인다(자식 구문별로 흩어지지 않음).
2. **Given** 룰 오너가 이름 없는 구문 노드, **When** 소비자가 오퍼레이션 이름·요약을 결정하면, **Then** PARENT_OF를 따라 **가장 가까운 루틴(라벨 ∈ 호출가능 루틴 집합)**을 찾아 그 루틴의 이름·요약을 쓴다.
3. **Given** 프로시저의 테이블 읽기/쓰기가 자식 구문(SELECT/UPDATE)에 붙은 그래프, **When** 소비자가 그 오퍼레이션의 테이블 영향을 조회하면, **Then** 자식들의 READS/WRITES를 **수집**해 누락 없이 제시한다.
4. **Given** framework 그래프(함수=오너, 자식 없음), **When** 동일 로직을 적용하면, **Then** 함수가 곧 가장 가까운 루틴이므로 동작이 바뀌지 않는다(0칸 상승 = 무변화).

### User Story 5 - 프론트엔드 표시 정합 (Priority: P2)

분석 그래프 표시(robo-data-frontend)는 대체로 현행 계약(단일 antlr-타입 라벨·`id`·프로시저→자식 트리)에 이미 정합되어 있으나, 두 지점이 어긋난다: (a) architect의 인제스천 모달이 비즈니스로직 개수 통계를 **옛 PascalCase 키**(`Rule/Example/Question`)로 읽어 생산자의 `RULE/EXAMPLE/QUESTION`과 안 맞아 **통계가 조용히 0**으로 표시되고, (b) 노드 상세에서 **프로시저(루틴)를 선택하면 그 룰이 안 보인다**(룰이 자식 구문에 붙어 있어서) — US4의 "오퍼레이션 단위 복원"이 표시에도 적용되어야 한다.

**Why this priority**: 데이터 흐름 자체를 막진 않으나 사용자가 보는 화면(통계·노드 상세)에서 정보가 빠져 신뢰를 떨어뜨린다.

**Independent Test**: dbms 분석 후 인제스천 모달의 Rule/Example/Question 카운트가 실제 개수로 뜨는지, 프로시저 노드 상세에 그 하위 룰이 모여 보이는지 화면으로 확인.

**Acceptance Scenarios**:

1. **Given** 생산자가 `RULE/EXAMPLE/QUESTION` 라벨로 노드를 적재, **When** architect 인제스천 모달이 개수 통계를 표시하면, **Then** 세 종류 카운트가 실제 값으로 렌더된다(0/미표시 아님).
2. **Given** dbms 프로시저 노드를 노드 상세에서 선택, **When** 그 오퍼레이션의 룰을 표시하면, **Then** 자식 구문에 붙은 룰들이 그 프로시저 아래 모여 보인다.

---

### Edge Cases

- 생산자가 어떤 룰에 흐름(NEXT/BRANCH)을 아예 적재하지 않은 경우 → 흐름 배열은 정당하게 비어야 하며, 이는 "버그로 인한 빈 배열"과 구분되어야 한다.
- 외부 호출 stub처럼 생산자가 의도적으로 다중 라벨/대체 종류를 부여한 노드 → 소비자 라벨 정합이 이 의도적 예외를 깨지 않아야 한다.
- 생산자가 "빈값 미저장" 규칙으로 어떤 노드에 특정 속성을 생략한 경우(예: 분석 안 된 함수의 요약 부재) → 소비자는 속성 부재를 정상 상태로 처리하고, 부재를 데이터 손상으로 오해하지 않아야 한다.
- 소비자가 한 종류의 식별자(예: `id`)로 매칭하되 생산자 적재가 동명이인(같은 이름 다른 함수)을 허용하는 경우 → 식별은 이름이 아니라 생산자의 단일 식별자에 근거해야 모호성이 없다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 소비자의 모든 analyzer-그래프 읽기 지점은, 읽는 노드/엣지 속성명을 **생산자가 그 종류에 실제로 기록하는 속성명**에 일치시켜야 한다(MUST). 특히 룰 흐름(NEXT/BRANCH)·지역 룰 주소에 쓰이는 흐름 식별 속성이 생산자 기록명과 일치해야 한다.
- **FR-002**: 소비자는 함수/루틴 노드를, 생산자가 통일한 **단일 식별자(`id`)** 를 우선 근거로 식별·조회해야 한다(MUST). 생산자가 더 이상 기록하지 않는 레거시 식별 속성에 식별을 의존해서는 안 된다.
- **FR-003**: 소비자는 생산자가 **현재 내보내지 않는** 노드/엣지 속성을 읽어 의미 있는 값으로 기대해서는 안 된다(MUST). 그러한 읽기는 (a) 생산자가 실제로 내보내는 동등 정보로 대체하거나, (b) 의도적으로 제거해야 한다.
- **FR-004**: 소비자는 생산자가 **그래프에 생성하지 않는** 노드 라벨/엣지 종류를 향한 매칭(죽은 패턴)을 포함해서는 안 된다(MUST). 그러한 패턴은 제거하거나 유효 패턴으로 교체한다.
- **FR-005**: 소비자가 생산자 그래프에 대한 갱신(쓰기/SET)을 수행하는 경우, 대상 노드 매칭이 0건이면 **조용한 no-op으로 끝나서는 안 된다**(MUST) — 누락이 관측 가능하도록 경고/로그로 표면화한다(조용한 실패 금지).
- **FR-006**: architect가 보유한 그래프 소비 **계약 문서**는, 생산자의 실제 출력 계약 및 소비자의 실제 동작과 **일치하는 단일 권위 문서**로 정렬되어야 한다(MUST). 라벨·식별자·멤버십/흐름 엣지 명칭이 생산자·소비자·문서 셋에서 동일해야 한다.
- **FR-007**: 생산자가 노드 라벨을 단일 antlr-타입(UPPER_SNAKE)으로 부여하고 호출가능/그릇 등 **역할은 타입에서 파생**한다는 계약에 맞춰, 소비자는 역할 판단을 폐기된 역할-라벨이 아니라 타입(집합)에서 파생해야 한다(MUST). 생산자의 의도적 다중-라벨 예외(외부 호출 stub)는 보존되어야 한다.
- **FR-008**: 생산자가 옛 가드(선행조건)/분기 메타 속성을 폐기하고 흐름을 NEXT(메인 순서)·BRANCH(분기) 엣지 + `flow_id` 계층으로만 표현하므로, 소비자는 이 메타를 생산자가 실제 적재한 흐름 구조에서 **재구성**해야 한다(MUST): (a) 지역 식별자는 흐름 엣지/관계 속성의 실제 명칭에서 읽고, (b) **분기 부모(else-leg)는 들어오는 BRANCH 엣지의 부모 규칙에서 도출**하며, (c) **선행조건(guard)은 들어오는 NEXT 엣지의 직전 규칙에서 도출**한다(결정사항 — 생산자 모델상 가용한 정보로 기존 ES 분해 의미를 최대한 보존). 폐기된 옛 속성명 직접 읽기는 제거한다.
- **FR-009**: 본 변경은 **소비자(architect)에 한정**되며, 생산자(analyzer)의 출력 계약을 변경하지 않아야 한다(MUST).
- **FR-010**: 본 변경으로 인제스천/소비 동작이 바뀌므로, 정합 검증은 생산자가 적재한 **실제 그래프에 대해 소비를 재실행**하여 값이 채워지는지 확인하는 방식으로 이뤄져야 한다(MUST) — 단위 테스트만으로 "동작함"을 단정하지 않는다.
- **FR-011**: 소비자 정합은 **framework·dbms 두 전략 모두**에서 성립해야 한다(MUST). 전략에 따라 그래프의 노드 종류가 달라지므로(framework: 파일/클래스/메서드/함수 / dbms: 스키마/테이블/컬럼/프로시저/트리거/SQL 구문), 소비자의 라벨·속성 정합이 양 전략의 노드 종류를 모두 포괄해야 하며, 한 전략 전용 가정(특정 라벨·속성만 존재한다는 가정)을 두어서는 안 된다.
- **FR-012**: 소비자는 생산자의 **대소문자/명명 규칙**(노드 라벨 = UPPER_SNAKE 단일 antlr 타입, 속성 = 생산자 정의 표기)을 정확히 따라야 한다(MUST). PascalCase·camelCase 등 옛 표기로 라벨/속성을 참조해서는 안 된다(예외: 생산자가 의도적으로 부여하는 다중-라벨 stub, 그리고 **소비자 자신의 세션-스코프 산출 노드**(`:Rule {session_id}` 등 architect 자체 노드)는 본 작업 대상 아님 — analyzer-그래프 읽기만 정합 대상).
- **FR-013**: 소비자는 룰/예시/질문의 **오너 노드가 호출 불가능한 구문 노드일 때**(dbms 깊은 구조), 그 노드에서 **부모(PARENT_OF)를 따라 가장 가까운 호출가능 루틴**(라벨이 호출가능 루틴 집합 = FUNCTION/PROCEDURE/METHOD/TRIGGER에 속하는 노드)을 찾아 **그 루틴을 오퍼레이션 단위로** 삼아야 한다(MUST). 오퍼레이션의 식별·이름·요약은 이 루틴 노드에서 취한다. framework에서는 오너가 이미 루틴이므로 0칸 상승(동작 불변)이어야 한다(전략 분기 없는 단일 규칙).
- **FR-014**: 소비자가 한 오퍼레이션(루틴)의 **테이블 영향(READS/WRITES)**을 조회할 때, dbms처럼 그 영향이 **자식 구문 노드에 붙어 있으면** 자식들을 따라 내려가 **수집·집계**해야 한다(MUST). 루틴 노드 자신에만 붙은 것으로 가정해 자식의 테이블 영향을 누락해서는 안 된다.
- **FR-015**: 오퍼레이션 단위 복원(FR-013)은 **식별/그룹핑뿐 아니라 매칭 근거(요약)에도** 적용되어야 한다(MUST) — 룰을 업무(task)에 매칭할 때 쓰는 요약은 좁은 구문 요약이 아니라 **소유 루틴의 요약**이어야 한다(매칭 품질 보존).
- **FR-016**: **프론트엔드 표시**도 현행 계약에 정합되어야 한다(MUST): (a) architect 인제스천 통계 표시가 생산자 라벨(`RULE/EXAMPLE/QUESTION`)로 개수를 읽어 0/미표시가 되지 않아야 하고, (b) 노드 상세에서 **루틴(프로시저)을 선택하면 그 하위 구문에 붙은 룰들이 그 루틴 단위로 모여** 표시되어야 한다(FR-013의 표시판). robo-data-frontend의 그래프 렌더·계층 트리는 이미 정합된 부분은 변경하지 않는다(회귀 방지).

### Key Entities *(include if feature involves data)*

> 아래는 소비자가 의존하는 그래프 계약면의 **현재 확인된 실태(생산자 실제 출력 기준)** 다. 정합 대상의 구체적 좌표를 명세 수준에서 고정하기 위한 것이며, 구현 방법(개별 쿼리 수정)은 plan 단계에서 다룬다.

- **함수/루틴 노드**: 생산자는 단일 `id`로 식별, 라벨은 단일 antlr 타입(예: 메서드/프로시저/함수 등), 파일 위치는 단일 경로 속성으로 기록. 요약/시그니처류는 분석 단계에서만, 빈값이면 미저장. 소비자는 식별을 `id` 기준으로, 부수 속성은 생산자 실제 기록명으로 읽어야 함.
- **룰(Rule) 노드 + 함수↔룰 관계**: 룰은 함수 간 공유될 수 있어, "이 함수에서의 지역 식별자/흐름"은 **노드가 아니라 함수↔룰 관계의 속성**에 담긴다. 소비자의 지역 식별자·흐름 연결 읽기가 이 관계 속성의 **실제 명칭**과 일치해야 함(P1 핵심).
- **예시(Example) 노드**: 생산자가 기록하는 속성 집합으로 한정해 소비. 생산자가 내보내지 않는 플래그성 속성에 의존 금지.
- **컨테이너 노드(파일/클래스/인터페이스 등)**: 생산자는 단일 `id`·단일 타입 라벨·정규화된 부수 속성으로 기록. 소비자의 컨테이너 조회·폴백 경로가 폐기된 식별/라벨 속성을 쓰지 않아야 함.
- **멤버십/흐름 엣지**: 생산자는 통합 멤버십 엣지와 흐름 엣지를 특정 명칭으로 기록. 소비자 트래버설과 계약 문서가 동일 명칭을 써야 함.
- **오퍼레이션 단위(루틴)**: framework·dbms 공통의 "호출 가능한 루틴"(FUNCTION/PROCEDURE/METHOD/TRIGGER). 룰/예시/질문의 **논리적 오너**이자 소비자의 그룹핑·식별·요약·매칭 단위. framework에서는 룰 오너 노드 자체, dbms에서는 룰 오너(구문)에서 PARENT_OF 상위로 가장 가까운 루틴. 이름 있는 노드.
- **구문 노드(dbms 자식)**: 프로시저 본문이 분해된 SELECT/IF/LOOP/CALL 등. 고유 `id`·단일 타입 라벨은 있으나 이름은 `""`. 룰·테이블접근(READS/WRITES)이 실제로 여기 붙음. 표시는 `타입[시작라인]`. PARENT_OF로 루틴에 연결, NEXT로 형제 실행순서.
- **계약 문서**: architect가 보유한 소비자-측 그래프 계약 문서. 생산자 실제 출력 및 소비자 실제 코드와 일치하는 단일 권위 문서가 되어야 함. **오퍼레이션 단위 복원 규칙(루틴 라벨 집합·PARENT_OF 상승·테이블 자식 수집)도 이 문서에 포함**.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 생산자가 룰·흐름을 적재한 그래프에 대해 소비자 룰 인제스천을 재실행했을 때, 지역 룰 식별자 NULL 비율 0%, NEXT/BRANCH 흐름이 실재하는 룰에서 흐름 배열 비어있음 비율 0%.
- **SC-002**: 소비자의 analyzer-그래프 읽기 지점 전수에 대해, 읽는 속성 키가 생산자 실제 기록 키의 부분집합이 아닌 지점(=조용한 NULL 유발 지점) 0건.
- **SC-003**: 소비자에 생산자가 생성하지 않는 노드 라벨/엣지 종류를 향한 매칭(죽은 패턴) 0건.
- **SC-004**: 소비자가 생산자 그래프에 쓰는 갱신 중 매칭 0건일 때 조용히 끝나는(no-op) 지점 0건 — 전부 관측 가능(경고/로그).
- **SC-005**: 계약 문서의 라벨·식별자·멤버십/흐름 엣지 명세가 생산자 실제 출력 및 소비자 실제 코드와 100% 일치(문서-코드 대조 시 불일치 0건).
- **SC-006**: 본 변경 후 생산자(analyzer) 코드·출력 계약 변경 0건.
- **SC-007**: **framework·dbms 두 전략 각각**으로 분석·적재한 그래프에 대해 소비를 재실행했을 때, SC-001~SC-004가 두 전략 모두에서 충족(어느 전략에서도 0건 회귀).
- **SC-008**: 소비자 그래프 접점에 PascalCase/camelCase 등 생산자 현행 표기와 다른 라벨/속성 참조 0건(의도적 다중-라벨 stub·소비자 자체 세션노드 예외 제외).
- **SC-009**: dbms 분석 그래프에서, 한 프로시저의 모든 룰이 **그 프로시저 1개 오퍼레이션으로** 묶이는 비율 100%(이름 없는 구문 단위로 흩어지는 룰 0건). 그 오퍼레이션의 테이블 영향 = 자식 READS/WRITES 합집합과 일치(누락 0).
- **SC-010**: dbms 인제스천 후 architect 인제스천 모달의 Rule/Example/Question 카운트가 실제 그래프 개수와 일치(0/미표시 아님). 프로시저 노드 상세에 하위 룰이 그 루틴 단위로 표시됨.

## Assumptions

- 생산자(analyzer)의 현재 출력 계약(단일 `id` 식별자, UPPER_SNAKE 단일 타입 라벨, 통합 멤버십 엣지, "빈값 미저장" 속성 규칙, 룰 흐름/지역 식별자를 함수↔룰 관계 속성에 보관)은 **고정 기준**이며 본 작업에서 변경하지 않는다.
- 소비자에 이미 진행된 라벨/식별자 마이그레이션(미커밋 진행분)은 **유효한 출발점**으로 유지·계승하며, 그 위에서 잔여 속성명 정합을 완성한다.
- 검증은 생산자가 적재한 실제 그래프에 대해 소비를 재실행하는 방식으로 수행 가능하다(전체 스택/실데이터 기동 전제).
- architect의 계약 문서는 현재 spec 036 산하에 존재하나, 본 작업에서 어느 위치를 단일 권위 문서로 둘지는 plan 단계에서 확정한다(기존 위치 갱신 또는 044 산하 신설).
- 본 작업은 소비자 인제스천 동작을 바꾸므로, 정합 효과를 보려면 대상 프로젝트의 **재인제스천**이 필요할 수 있다.
