# 레거시 근거 조회 계약

`robo-cluster` 도구가 주입되면 **탐색 → 심층 → 배분 → 갭마감** 4단계를 수행한다.
그래프 존재 여부를 추측으로 생략하지 말고 `cluster_retrieve` 의 실제 결과로 판정한다.

> **이 계약의 핵심은 검색량이 아니라 배분이다.** 실측 결과 근거 누락의 93%는 "검색을 안 해서"가
> 아니라 **"이미 검색해 놓고 요소에 붙이지 않아서"** 발생했다(후보 122개를 확보한 실행에서
> 요소 11개가 빈 채로 나옴). 그러니 호출을 늘리는 데 힘을 쓰지 말고, **확보한 후보를 끝까지
> 쓰는 데** 힘을 써라.

---

## S1. 탐색 — 축을 훑는다

`mcp__robo-cluster__cluster_retrieve` 를 업무 축마다 1회 호출한다. 응답은 후보의
ID·이름·요약·관계 단서·RULE/EXAMPLE 목록이며 원문 코드와 전체 컬럼은 없다.

축은 아래를 참고해 뽑되, **한 질의가 여러 축을 덮으면 합쳐도 된다**(질의 수를 늘리는 것이
목적이 아니다).

- **주 동작**: 요구가 직접 말하는 행위
- **대칭 동작**: 되돌림·반대(적립↔사용, 생성↔취소, 예약↔해제)
- **생애주기 말단**: 만료·소멸·마감·정리·배치 — 자주 누락된다
- **상태·이력**: 상태 코드 전이, 이력 테이블
- **경계 인접**: 이 기능이 호출하거나 이 기능을 호출하는 옆 도메인

**질의는 한 번에 설계하라.** 축을 정한 뒤 질의 문구를 **한 묶음으로 먼저 만들고**, 서로 거의
같은 것은 합친 뒤 발행한다. 비슷한 질의를 연달아 던지면 같은 후보만 다시 온다.

**종료 조건 (상한이 아니라 이것이 주 기준)** — 아래 중 하나면 탐색을 멈춘다.
- **포화**: 방금 질의가 돌려준 후보의 **대부분이 이미 본 것**이면 그만한다. 보통 2~4회면 포화한다.
- **커버리지**: 만들 설계 요소 전부가 이미 대응 후보를 가졌다면 더 찾을 이유가 없다.

예산 감각: 축 수 + 2회 정도가 적정이고 **8회를 넘기지 않는다**(넘길 만하면 질의가 서로
비슷하거나 축이 잘못 잡힌 것이다). 결과가 비면 동의어·상위어·영/한을 바꿔 한 번 더 시도하고,
그래도 비면 그 사실을 narration 에 남기고 진행한다.

## S2. 심층 — summary 를 읽고 고른다

S1 후보의 **요약과 RULE 목록을 실제로 읽고**, 아래에 해당하는 것만
`mcp__robo-cluster__node_detail(node_id)` 로 조회한다. 전부 기계적으로 조회하지 않는다.

- 이 변경이 **바꾸거나 대체할** 현행 구현
- 불변식·상태 전이·검증 규칙의 원천이 될 요약을 가진 것(RULE 보유 노드 우선)
- 트랜잭션 경계 판단에 필요한 것(무엇과 무엇이 한 커밋인지)

원문 코드·파일·시작/끝 줄·실제 관계·TABLE 컬럼은 **상세 응답만** 근거로 쓴다. 요약만 보고
원문·라인을 보았다고 주장하지 않는다. 상세 오류가 나면 그 ID 를 근거로 확정하지 않는다.

## S3. 배분 — 후보를 요소에 붙인다 (이 계약에서 가장 중요)

설계 요소 초안을 만든 뒤, **S1/S2 로 확보한 후보 목록 전체를 다시 한 번 훑으면서**
각 요소에 근거를 배분한다. **추가 도구 호출은 필요 없다** — 이미 받은 결과를 쓰는 것이다.

훑는 방향은 **요소 → 후보**가 아니라 **후보 → 요소**로 한다. 즉 "이 요소의 근거가 뭐지?"가
아니라, 후보를 하나씩 보며 **"이 후보는 내가 만든 어느 요소를 뒷받침하는가?"** 를 묻는다.
전자는 잊고 넘어가기 쉽고, 후자는 확보한 후보를 빠짐없이 소진하게 한다.

배분 시 주의:
- **이름이 아니라 요약으로 판단하라.** 레거시 이름(`settlement_close`)과 설계 이름
  (`CloseSettlement`)은 언어가 다르다. 요약이 "정산 건을 마감 처리한다"면 그것이 근거다.
- 상위 축 질의로 나온 후보를 **하위 요소(Command·Event·ReadModel)에 배분하라.** 상위
  Aggregate 에만 붙이고 끝내지 않는다.
- 한 후보가 여러 요소를 뒷받침하면 **여러 요소에 붙인다**(중복 배분은 정상이다).
- 배분할 곳이 없는 후보는 그냥 두면 된다(검색은 넓게, 배분은 정확하게).

## S4. 갭 마감 — 그래도 빈 요소만

S3 후에도 `legacyRefs` 가 빈 요소가 남으면, **그 요소에 한해** 이름·책임으로
`cluster_retrieve` 를 1회 더 호출한다(예: `ExpirePoints` → "포인트 소멸 만료").
새 후보가 나오면 배분하고, 없으면 그때 `[]` 로 확정한다.

**빈 배열에는 입증 책임이 있다.** `legacyRefs: []` 는 "안 찾아봤다"가 아니라
"S3 배분과 S4 재검색을 거쳤는데 없다"를 뜻한다.

---

## 검색과 검토의 구분

검색만 한 노드는 "후보", `node_detail` 까지 성공한 노드는 "검토함"이다. 도구 호출이 실패하거나
결과가 비면 그 사실을 narration 에 남기고 진행할 수 있지만, 레거시 구현을 요구사항보다
우선하거나 그대로 복사해서는 안 된다.

## 완료 게이트

- 도구가 주입된 실행은 `cluster_retrieve` 시도 1회 이상이 없으면 완료가 아니다.
- 검색 결과에 후보 ID 가 하나라도 있으면 `node_detail` 성공 1회 이상이 있어야 한다.
- **S3 배분을 수행하지 않았으면 완료가 아니다.**
- 후보가 없거나 모든 상세가 실패한 경우에만 그 사실을 명시하고 진행한다.

## narration 보고 (필수 2줄)

```
[레거시 참조] <검토한 핵심 함수·테이블과 줄 범위>
[근거 커버리지] 후보 N개 → 요소 M개 중 K개 연결 · 미연결 L개(S4 재검색 완료)
```

## 요소별 근거 기록(legacyRefs)

확보한 근거는 narration 으로 끝내지 않고 **최종 JSON 의 각 요소에 `legacyRefs` 로 기록한다**
(형상·불변식: strategic 요소는 `strategic-output-schema.md`, tactical 요소는
`output-schema.md`). 검색·검토하지 않은 ID 를 기록하면 서버 검증이 제거하고 경고를 남기므로,
확인한 ID 만 사용한다.
