# 레거시 근거 조회 계약

Proposal 실행에 `robo-cluster` 도구가 주입되면 설계 전에 다음 두 단계를 반드시 시도한다.
그래프 존재 여부를 추측해서 생략하지 말고 `cluster_retrieve`의 실제 결과로 판정한다.

1. `mcp__robo-cluster__cluster_retrieve`를 핵심 업무 축마다 1회(전체 1~3회) 호출한다. 응답은
   후보의 ID·이름·요약·관계 단서·RULE/EXAMPLE 목록이며 원문 코드와 전체 컬럼은 없다.
2. 목록에서 설계 판단에 실제로 필요한 ID만 고르고
   `mcp__robo-cluster__node_detail(node_id)`을 각각 호출한다. 원문 코드·파일·시작/끝 줄·
   실제 관계·TABLE 컬럼은 이 상세 응답만 근거로 사용한다. 후보 전부를 기계적으로 조회하지 않는다.

검색만 한 노드는 “후보”, `node_detail`까지 성공한 노드는 “검토함”이다. 원문·라인·컬럼을
상세조회하지 않았으면 보았다고 주장하지 않는다. 상세 오류가 나면 해당 ID를 근거로 확정하지 않고
오류 사실을 narration에 남긴다. 도구 호출 자체가 실패하거나 검색 결과가 비면 그 사실을 narration에
남기고 레거시 근거 없이 진행할 수 있지만,
레거시 구현을 요구사항보다 우선하거나 그대로 복사해서는 안 된다.

완료 게이트는 다음과 같다. 도구가 주입된 실행은 `cluster_retrieve` 시도 1회 이상이 없으면 완료가 아니다.
검색 결과에 후보 ID가 하나라도 있으면 판단에 필요한 ID를 골라 `node_detail` 성공 1회 이상이 있어야
완료다. 후보가 없거나 모든 상세가 실패한 경우에만 오류·빈 결과를 명시하고 다음 단계로 진행한다.

최종 narration에는 `[레거시 참조] <검토한 핵심 함수·테이블과 줄 범위>`를 한 줄로 남긴다.

## 요소별 근거 기록(legacyRefs)

검토·검색으로 확인한 근거는 narration 으로 끝내지 않고 **최종 JSON 의 각 요소에
`legacyRefs` 로 기록한다**(형상·불변식: strategic 요소는 `strategic-output-schema.md`,
tactical 요소는 `output-schema.md`). 요소를 만들 때
"이 요소는 어느 검색/검토 노드에서 나왔는가"를 스스로 물어 해당 `nodeId` 를 달고,
대응 근거가 없는 신규 요소는 빈 배열로 남긴다. 검색·검토하지 않은 ID 를 기록하면
서버 검증이 제거하고 경고를 남기므로, 확인한 ID 만 사용한다.
